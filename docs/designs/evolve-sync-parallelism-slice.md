## 0. When parallelism wins, and when it does not (read this first)

**This is the most important section in the doc, and it is deliberately blunt. The technique is built for ONE regime — slow evaluation — and the AlphaEvolve paper says so almost verbatim.** That paper states it is feasible to spend on the order of **100 compute-hours to evaluate a single solution**, and that *"unless individual evaluations are parallelized to reduce their wall-clock duration, this can slow down the rate at which new generations appear … evaluation is embarrassingly parallel … allowing AlphaEvolve to distribute this work through asynchronous calls to an evaluation cluster"* (`/tmp/alphaevolve.txt:420-426`). Table 1 in the same paper draws the line directly: FunSearch *"needs fast evaluation (≤20min on 1 CPU)"*, whereas AlphaEvolve *"can evaluate for hours, in parallel, on accelerators"* (`:84`). **Parallel evaluation is not a hedge for this design — it is the operating regime the whole technique was built for.** Circle-packing's short scipy optimize is the opposite regime (fast, uniform eval); it is the `workers:1` regression fixture (gate item 1), **not** the speedup demo.

The realistic value is therefore *real but bounded*, and the bound has two distinct origins for two distinct domain classes: (i) for **CPU-light slow-eval** domains, the bound is the serialized-tool-call floor plus the straggler tax (§0.1); (ii) for **GPU/TPU/memory-bound** domains, the bound is *resource contention inside the single shared container* — the off-mutex-overlap win evaporates because N evals serialize on one accelerator or OOM each other (§0.3, the biggest gap this analysis found). Here is the honest accounting for both.

### 0.1 What actually runs concurrently — and what does not (the CPU-light case)

Three facts bound the realized speedup. They apply with full force only to **CPU-light, memory-bounded** evals (the regime where N execs into one container genuinely overlap); the GPU/TPU-bound case is governed by the resource model in §0.3, where the win evaporates for a different reason.

1. **Every agent tool call is serialized across all lanes by the coordinator's FIFO mutex — but for the target domains this is negligible, not dominant.** `handleToolCall` and `handleStructuredToolCall` run their *entire* body — policy eval, audit append, circuit-breaker check, **and the escalation await** — inside `this.callMutex.withLock(...)` (`tool-call-coordinator.ts:310-313`, `:348-357`); the mutex is fair FIFO and non-reentrant (`async-mutex.ts:13`, `:15`, in `src/trusted-process/`). So N `researcher`/`analyzer` lanes do **not** make their filesystem/MCP tool calls concurrently — every tool touch queues single-file. **But weigh this honestly against the real domains.** Production evolve agents are *think-then-write-one-diff*: ASI-Evolve's researcher uses diff-based evolution, editing a sampled parent via a handful of SEARCH/REPLACE edits (`config.yaml:44-49`), so a lane's tool-call footprint is a small, bounded number of file touches. Against a per-candidate eval measured in *compute-hours* (`/tmp/alphaevolve.txt:420-421`), the milliseconds those serialized touches cost are noise. **The mutex floor is why N=1 circle-packing is a wash (fast eval, so the floor is a visible fraction); it is NOT the dominant concern for the slow-eval domains the slice targets.** The agents *think* (LLM generation) in parallel; their *tool calls* serialize, and for the target domains that serialization is irrelevant against the eval cost. (The agent-path *escalation* stall, §5.2, is a separate latency-cliff that stays in-scope regardless of domain.)
2. **The expensive deterministic steps DO run concurrently — and this is the real win for CPU-light evals.** `sample`, `evaluate` (the evaluator subprocess), and `analysis_record` are trusted `bundle.docker.exec` calls (`orchestrator.ts:2667`), explicitly **not** coordinator tool calls (`src/trusted-process/CLAUDE.md`, "Deterministic container exec"). They never touch the `callMutex`. So the slow per-lane work — the evaluator — overlaps across lanes **when it is CPU-light/memory-bounded**. **The slow-evaluate-stalls-all-lanes fear is therefore wrong** (CONSIDER-1): `evaluate` never holds the mutex. *Caveat made load-bearing in §0.3:* "overlaps" assumes the evals do not contend on a shared scarce resource (cores, RAM, a single GPU). For the GPU/TPU heavy hitters that assumption fails and N concurrent execs serialize on the accelerator — see §0.3.
3. **The barrier is a hard `Promise.all` join (§6.2), so wall-clock per batch is the straggler lane's time, not the mean — and for real domains this is the dominant tax.** LLM-latency variance across lanes makes the straggler tax real, and it grows with N. For the target domains it is *large*: real evaluation is high-variance (cascade evaluation prunes weak candidates early while survivors run orders of magnitude longer; ASI-Evolve's v2 carries 300s eval / 1800s engineer timeouts, `config.yaml:20`,`:61`), so the straggler-to-mean ratio is wide. The hard barrier pays that ratio every batch. This is the price the sync model accepts as a stepping stone (§13); AlphaEvolve avoids it by being async by construction (`:455-462`).

Putting (1)–(3) together, the realized speedup obeys an Amdahl bound, not a linear-in-N one:

```
speedup(N)  ≈  1 / ( f_serial  +  f_parallel / N )   ×   (1 / straggler_inflation(N))

  f_serial   = fraction of per-round wall-clock spent in serialized agent tool calls
               (every researcher/analyzer FS/MCP touch) + the post-barrier serialized
               join (cognition promotion + stop-signal recompute, §9)
  f_parallel = fraction in genuinely-concurrent work: in-agent LLM generation +
               off-mutex evaluator docker.exec
  straggler_inflation(N) ≥ 1, grows with per-lane latency variance
```

The honest reading: **the win scales with `f_parallel`, which is dominated by `evaluate` + LLM-thinking time.** When evaluation is slow (the paper's ~100 compute-hours per candidate, `/tmp/alphaevolve.txt:420-421`), `f_parallel → 1` and the speedup approaches `N / straggler_inflation` — a real, large win, **provided the N concurrent evals actually overlap.** That proviso is the whole content of §0.3: the Amdahl model above silently assumes `evaluate`'s N concurrent execs run in parallel, which holds for CPU-light/memory-bounded evals but *fails* for GPU/TPU/memory-bound evals that serialize on one accelerator or OOM each other (`:2180`). When evaluation is fast and uniform (circle-packing), `f_parallel` shrinks, the serialized-tool-call floor and the straggler tax dominate, and the speedup collapses toward 1.

### 0.2 Why circle-packing is the wrong motivating domain

- The circle-packing evaluator is a short scipy/SLSQP optimize — fast and CPU-bound. N concurrent evals on **one shared container** (`sharedContainer: true`, `workflow.yaml:8`; N concurrent `docker.exec` into one `containerId`, `orchestrator.ts:2667`) **contend for the same cores** rather than overlapping I/O. So even the off-mutex win (fact 2) erodes: fast CPU-bound evals do not parallelize on a contended container.
- Token spend is **strictly N×** (N researcher + N analyzer agent runs per batch).
- The search-quality slice's own framing is that circle-packing's win comes from *cognition + multi-parent recombination* (`evolve-search-quality-slice.md`), **not** from raw round throughput.

**Conclusion the doc commits to:** sync parallelism is a win for **slow-eval / high-per-round-latency domains** — the regime the AlphaEvolve paper says the technique was built for (`/tmp/alphaevolve.txt:420-426`, `:84`) — *provided the evals overlap in the shared container* (§0.3). It is roughly a **token-for-throughput wash, plausibly a net loss, for fast-uniform-eval domains like circle-packing** (where evals contend on cores, tool calls serialize, and you pay full N× tokens for a sub-N throughput gain). **The slice's value must be validated on a slow-eval workload that is also CPU/memory-bound (so it overlaps in one container); the natural target is ASI-Evolve's own LLM-serving scheduler (§14 OQ6). Circle-packing is the backward-compat fixture (`workers: 1`, gate item 1), not the speedup demo.** Phase 0 (§14) measures the per-round tool-call fraction and evaluator-cost fraction so this is grounded, not asserted.

### 0.3 Resource model and its limits — where the off-mutex win evaporates (the biggest gap)

§0.1 fact 2 says the off-mutex `evaluate` execs "overlap across lanes." **That is only true when the evals are CPU-light and memory-bounded.** The design runs all N evals as N concurrent `docker exec` into ONE shared container (`sharedContainer: true`, `workflow.yaml:8`; N concurrent `bundle.docker.exec(bundle.containerId, …)`, `orchestrator.ts:2667`). Overlap of N processes in one box requires that the resource each eval is bound on is *not* a single shared scarce resource. For the AlphaEvolve flagship domains, it is exactly that:

- **Matrix-multiplication / tensor-decomposition** (`/tmp/alphaevolve.txt:123`, full results `:2170-2181`) — GPU-bound. The paper's smoking gun is explicit: running discovered programs *"on sizes beyond ⟨5,5,5⟩ on 1000 random seeds on evaluators with a single GPU accelerator, we often run out of memory"* (`:2180`). N concurrent evals on one GPU do not overlap — they serialize on the accelerator or OOM each other.
- **Training-kernel / FlashAttention tiling** (`:124`, *"optimizing matrix-multiplication kernels used to train LLMs … optimizing the runtime of attention in Transformers"*; the Pallas-kernel tiling study minimizes *actual runtime on real TPU accelerators*, `:1235-1239`) — TPU-bound, and the *score itself is a wall-clock measurement*, so co-located evals would corrupt each other's timing even if memory allowed.
- **TPU arithmetic-circuit Verilog** (`:124`, *"optimizing arithmetic circuits used within TPUs"*) — bound on heavyweight synthesis/simulation tooling (memory + licensed simulators), again not freely co-locatable.

For all three, **N concurrent execs in one container do not overlap; the design's central off-mutex-overlap win (§0.1 fact 2) evaporates exactly where evaluation is most expensive** — the worst possible place to lose it. DeepMind's own resolution was **not** N execs into one box: it was a **fleet of evaluator nodes** behind the async controller (the pipeline is *"a controller, LLM samplers, and evaluation nodes … optimized for throughput,"* `:455-462`; the Borg-scheduler study evaluated candidates against *"snapshots of workloads and capacity across Google's fleet,"* `:1176`).

**Design implication (the bridge to distributed evaluator scheduling).** For the GPU/TPU/memory-bound domains, the shared-single-container model is insufficient and needs one of:

1. **Per-lane resource isolation — but CPU/mem and GPU are NOT equally available (DA-S1).** cgroup **CPU/memory** limits per exec (`--cpus`/`--memory`) are real and per-container *today*, so a fat-memory lane can be capped not to OOM its CPU/RAM-bound peers, on localhost, now. **GPU isolation is a separate, harder, hardware-dependent problem:** MIG partitioning is datacenter-GPU-only (A100/H100-class) and **statically pre-partitioned** (no on-demand per-lane carving); GPU time-slicing gives **no memory isolation**, so it does NOT fix the `:2180` VRAM OOM (time-sliced processes still share the GPU's memory). So this option delivers CPU/mem isolation cheaply but GPU isolation only on specific hardware with static partitioning — do not list them as one knob. With real per-device isolation, N caps at the real device count.
2. **N containers (one per lane)** — abandons `sharedContainer` for these domains; each lane gets its own container with its own device, at the cost of the shared `/workspace`/DB convenience (the DB is already cross-process safe, §8, so this is mechanically feasible — but it is a topology change, not a tweak).
3. **A host-side bounded eval queue** — keep the shared container for the agent/sample/record work, but route `evaluate` through a bounded pool sized to the device budget, so at most `devices` evals run concurrently and the rest queue. This is the smallest step toward the paper's evaluator cluster.

**This is the bridge to distributed evaluator scheduling, and §8.5 makes the bridge concrete via a seam that already exists.** The orchestrator never creates a container directly — it calls the injected `createWorkflowInfrastructure` factory (`orchestrator.ts:463`, called `:862`; default localhost impl `:1216`) and then only touches the bundle through `bundle.docker.exec` (`:2667`). "The FSM talks to bundles, not nodes." All three options above are therefore expressible as **a different infrastructure-factory implementation** that places eval slots with per-slot resource budgets — **cgroup CPU/mem caps are available per-container today; GPU isolation is a harder hardware-dependent follow-up, NOT the same knob** (DA-S1, §0.3 option 1) — **localhost first** (the same-host emulation, where the current shared container *is* option 0), distributed later. This is explicitly out of scope for *this* sync slice's first cut, but it is the load-bearing reason the slice's validated win (§14 OQ6) must be a CPU+memory-bound domain (where the shared container suffices), and the reason the GPU/TPU flagship domains are flagged as needing this resource model before the technique delivers there. The sync barrier and the lane machinery (§6) are unaffected by which of 1–3 is chosen; the resource model is orthogonal to the FSM topology and slots in at the `evaluate` exec point (`orchestrator.ts:2667`) through the scheduler factory (§8.5).

### 0.4 Knock-on consequences of the serialized tool calls

- **Circuit-breaker false-trips become a correctness risk, not just tuning** (CONSIDER-5). With identical-parent lanes (§7.4) reading the same cognition/candidate files, the `(tool, argsHash)` calls are *identical*, not merely similar. N lanes × repeated identical reads can trip the default 20/60s window and **deny** legitimate lane work mid-batch. So the breaker window must be made `workers`-aware **by default when `workers > 1`** (not optional config) — see §5.2 item 3.
- **The slowest realistic risk is the agent-path escalation stall** (§5.2, §10), where one lane's policy escalation holds the FIFO mutex across the full human-response latency, freezing every other lane's tool calls. That is a *latency cliff*, not a throughput gradient, and it is addressed as a Phase 0 gate item.

## 1. Summary & goals

The shipped linear round-chain runs exactly one candidate per round: the orchestrator emits `design`, the chain `sample → … → analysis_record` produces and records one node, and control returns to the orchestrator, which re-decides continue/complete/escalate. ASI-Evolve's reference pipeline runs **N workers per round** (`pipeline.parallel.num_workers`, "2–4 for production"; `scripts/README.md:286`) — N candidates explored concurrently against one shared population DB, joined per round before the controller advances.

This slice adds that fan-out, **synchronously**: at `design`, the orchestrator spawns **N concurrent round-chains** (lanes), waits at a barrier until **all N** have recorded their node, then re-decides continue/complete/escalate **once per batch**. N is a configurable worker count; **`workers: 1` must reproduce today's single-worker behavior byte-for-byte** (the default).

**Read §0 before anything else.** The technique is built for **slow evaluation** — the AlphaEvolve paper's ~100-compute-hour-per-candidate regime, where *"unless individual evaluations are parallelized … evaluation is embarrassingly parallel"* (`/tmp/alphaevolve.txt:420-426`) and AlphaEvolve *"can evaluate for hours, in parallel, on accelerators"* (`:84`). The realized speedup is still bounded below N, and for two distinct reasons by domain class: (i) for CPU-light slow-eval domains, the bound is the (negligible, against a 100-hour eval) serialized-tool-call floor plus the straggler tax; (ii) for the GPU/TPU/memory-bound flagship domains (matmul kernels, FlashAttention tiling, TPU circuits), the bound is **resource contention inside the one shared container — the off-mutex evaluator overlap evaporates** because N evals serialize on one accelerator or OOM each other (§0.3, the paper's own *"we often run out of memory"*, `:2180`). "N candidates explored concurrently" is true only for the off-mutex deterministic work (`evaluate`) and the in-agent LLM generation — and only when that `evaluate` does not contend on a shared device. This slice's value is the off-mutex evaluator overlap on **CPU/memory-bound** slow-eval workloads; the GPU/TPU domains need the resource model in §0.3 first. Circle-packing is the `workers:1` regression fixture, not the speedup demo.

The defining constraint is **single-active-state**: the FSM is single-active-state by construction — `String(snapshot.value)` (`orchestrator.ts:1873`), single `instance.currentState` (`:1916`, `:1510`), single `activeGateId` (`:2690`), single `currentPersonaByBundle` entry per bundle (`:1037`), flat `transitionHistory` (`:1890-1897`). We do **not** use XState **parallel REGIONS** (a `type:'parallel'` value is an object → `String(value)` becomes `"[object Object]"`, breaking the spine). The parent FSM stays in **one real `workers` state** (shape A, §4.2) whose invoke body (`runFanOutSegment`, itself a `fromPromise` actor — plain async TS, not XState transition land) **instantiates N free-standing child round-machine actors** via `createActor(roundMachine, { input })` + `.start()` and barrier-joins them with `Promise.allSettled` over per-child subscribe-until-`done` promises (§6 — the hard part; FSM-M2). These are **free-standing actor instances, NOT in-context `spawn`**, so there are no parent-visible `onDone` outputs to consume (unlike `invoke`, whose done is consumed at `:1603`); the join reads each child's final-state `output` off its own snapshot. **Honest framing (CHANGE 1):** single-active-state is preserved because the children are runtime actor refs the invoke body holds, *not* regions of the parent's `value`. Each child is a real actor, so per-lane visit counts / `maxVisits` / token+stall write-back fire **natively from the child's own context — no manual re-implementation, within a batch** (§7.5); under the first-cut DB-as-truth resume they **reset on re-spawn**, so per-lane `maxVisits` is not crash-durable (§6.1/§6.3). The residual costs are **observability fan-in** — per-lane transition history, *live token accounting* (a correctness issue, FSM-S2), the lifecycle bus, and `messageLog` writers (§6.3/§7.5) — and the **round-limit coverage regression** (§4.4/§7.5). We do not present those as free.

The whole slice is six files:

- **`workflow.yaml`** — add **one real fan-out `workers` state** (shape A, §4.2) that spawns the round child machine; keep the five spoke states *declared* (prompts/run authored there) and marked `fanOutMember` so they become the child machine's states; add a `workers` count to `settings`; add a `schedule: { pool: eval, resources }` marker on `evaluate` (§8.6); re-point the orchestrator's `design` edge at `workers`. The gates and terminals are otherwise unchanged.
- **`evolve_result.py`** — the bridge gains a per-lane working-area discipline (§7: each lane gets its own `current/lane_<k>/` sub-path instead of the single `current/`), barrier-side `sample(n=N)` lane divergence + content-hash-at-fan-in (§7.4), a barrier-side serialized cognition promotion (§9), and aggregated-escalation handling (§10).
- **`machine-builder.ts`** — factor the round sub-chain (`sample → researcher → evaluate → analyzer → analysis_record`) into a **child machine definition** (`setup({ actors }).createMachine(...)`) instantiated per lane (free-standing `createActor`, §6.2), with the same `agentService`/`deterministicService` actors **re-provided per `createActor`** (the `roundMachine` is a *different* machine — FSM-S4), plus `recorded`/`blocked`/`errored` final states (§6.1, FSM-S5). The parent machine gains the `workers` state.
- **`orchestrator.ts`** — `runFanOutSegment` (the `workers`-state invoke, a `fromPromise` body): **mint the scope bundle once before fan-out** (§6.2, fixes the unguarded mint race), barrier-side `sampleLaneInputs` (§7.4), **`createActor(roundMachine, { input })` + `.start()` per lane**, **barrier-join via `Promise.allSettled` over per-child subscribe-until-`done` promises** (§6, FSM-M2/S5 — NOT in-context `spawn`, NOT `onDone` outputs), and route the child's `evaluate` through the eval-slot pool (a thin trusted-exec target, NOT a `DockerInfrastructure` bundle — DA-M1, §8.5/§8.6). Per-lane bookkeeping is the child's own context within a batch (resets on resume, §6.1) — no manual threading (§7.5). Plus the §5 coordinator-concurrency audit + escalation-mutex fix and the §5.4 orchestrator-side `instance.*` mutation audit (FSM-S4).
- **`validate.ts` / `lint.ts`** — schema additions for `settings.workers`, `fanOut`, `fanOutMember`, `segment`, and `schedule` (§8.6) (these are silently stripped by default `.strip()` Zod objects unless added — SF1), plus the lint/validator rules (§4).
- **`types.ts`** — the `workers` settings field and the new state-definition / transition fields, with the dead `maxParallelism`/`PARALLEL_*` graveyard reconciled (§4.5, CONSIDER-2).

### Non-goals (out of scope)

- **Async / streaming parallelism is OUT.** No controller-tick-on-record, no bounded worker pool that launches lane K+1 the moment lane K records, no mid-batch drain pipelining. The barrier is a hard join: all N lanes complete before the orchestrator re-decides. §13 maps which pieces are async-extensible so the follow-up is additive.
- **No `evolve_core/` change.** The byte-verbatim invariant holds (§12). The engine's durable DB is already cross-process safe (§8); the cognition store's non-safety is handled by serializing promotion at the barrier (§9), a bridge-side change.
- **No XState parallel REGIONS (`type:'parallel'`) on the parent.** That is the mechanism that breaks `String(snapshot.value)` (§6.4) and stays forbidden. **Free-standing child ACTORS (`createActor` + `.start()`, instantiated inside the `workers` invoke body) ARE used** (CHANGE 1) — a *different* mechanism that preserves single-active-state (children are runtime refs the invoke body holds, not regions of the parent's `value`; §6.4). The parent stays single-active-state in `workers`; the concurrency is N free-standing child round-machine actors. (In-context `spawn` is *not* used in the first cut — it would only become relevant if/when the checkpoint migrates to `getPersistedSnapshot()` to make in-flight children crash-durable, §6.3/FSM-M2.)
- **No new human-gate, terminal, or persona.** The gates (`preflight_review`, `human_escalation`, `final_review`) and terminals (`done`/`failed`/`aborted`) are unchanged. Lanes are homogeneous (`global` persona) so no per-lane policy swap (§8).
- **No change to N being chosen per-round.** `workers` is authored once in `settings` (or `preflight`), fixed for the run, like `maxRounds`. Per-round adaptive N is a later concern.

### Goals (binary, the gate — §15)

After this slice:

1. **`workers: 1` is a no-op.** A `workers: 1` run produces a byte-identical `nodes.json` (modulo `created_at`) to the pre-slice linear chain on the same seeded inputs. This is the backward-compat gate.
2. **N lanes record N nodes per batch.** A `workers: 3` Docker integration run records 3 nodes per batch (node count advances by 3 between orchestrator entries), each with a distinct `step_name`, no collision, no lost lane.
3. **The barrier joins.** The orchestrator is entered exactly once per batch (not once per lane); a unit/integration test proves the `workers` state's invoke resolves only after all N lane traversals have recorded.
4. **Coordinator survives N concurrent streams.** A test drives N concurrent agent tool-call streams through one `ToolCallCoordinator` and asserts the audit log has no interleaved/torn entries and the circuit breaker counts are correct (§5).
5. **Aggregated escalation.** A `workers: 3` run where one lane hits `evaluator_blocked` drains the batch and opens exactly **one** `human_escalation` gate (single `activeGateId` preserved; §10).
6. **Resume re-runs the in-flight batch idempotently.** A simulated crash mid-batch, then `workflow resume`, re-runs the batch and leaves no duplicate nodes (§11).
7. **`tsx src/cli.ts workflow lint src/workflow/workflows/evolve/workflow.yaml --strict` is clean**, and the existing evolve gates still pass.

## 2. Background — the linear chain and why sync

### 2.1 The shipped linear round-chain

Today (`workflow.yaml`): `orchestrator` (`:303`) emits `design` → `sample` (`:362`, deterministic, `--n-from-spec`) → `researcher` (`:390`, agent) → `evaluate` (`:424`, deterministic) → `analyzer` (`:450`, agent) → `analysis_record` (`:474`, deterministic, the single durable engine node-write) → back to `orchestrator` (`:497`). Each round appends exactly one node. The orchestrator reads `database_data/nodes.json` + `current/stop_signals.json` and routes `design`/`complete`/`evaluate`/`escalate` (`:338-360`).

The per-round scratch lives in a single `current/` directory (`evolve_result.py:54-55`), cleared at the top of `sample` (`_clear_current_round`, `:58-72`). `step_name` is derived from the global node count: `step_name = f"step_{done_rounds + 1:04d}"` where `done_rounds = _node_count(run_dir)` (`:575-576`). **This single-`current/`, count-derived-`step_name` design is the thing N concurrent lanes break** — §7 resolves it.

### 2.2 Why synchronous (settled — not relitigated here)

The sync (barrier-batched) model was chosen over async in the prior human discussion. The load-bearing reasons it is *implementable now*, restated as grounding (not as an argument to reopen):

- **The barrier is a clean single-active-state checkpoint boundary.** Between batches the FSM is in exactly one state (`orchestrator` or the `workers` state) with a flat checkpoint — no per-lane logical FSM state to serialize. Async would need the FSM to represent "lane 2 recorded, lanes 1 and 3 still evaluating," which the single-active-state spine cannot express without the vestigial parallel scaffolding the codebase already retired (`types.ts` `PARALLEL_*` events, `parallelResults`; lint WF005 retired, `lint.ts:43-45`).
- **Homogeneity buys safety.** The N lanes are identical (`global` persona), so the per-state policy swap (`cyclePolicy`, `orchestrator.ts:988`) happens **once before the batch** — no swap race (§8). Async with heterogeneous lanes would reintroduce it.
- **The engine DB is already barrier-friendly.** Concurrent `record`s are safe (§8); serializing the `sample`+`record` critical section through the join keeps DB-mutation order deterministic while `design`/`eval` run in parallel (§12).

## 3. The fan-out / barrier-join model

```
                      orchestrator (single FSM state, single-active)
                            │  verdict: design  (workers = N)
                            ▼
                ┌───────────────────────────────┐
                │   workers  (ONE FSM state)     │   ← invoke body creates N child actors
                │                                │
   FAN-OUT  ──▶ │  lane 0   lane 1   …  lane N-1 │   N free-standing child round-machine actors
                │   │         │            │     │   (createActor + start), concurrent, over ONE container
                │  sample    sample      sample │   (each: sample→researcher→
                │  research  research    research│    evaluate→analyzer→record)
                │  eval      eval        eval    │
                │  record    record      record │   ← per-lane node append (DB-safe)
                │   └────┬────┴─────┬──────┘     │
                │        ▼ BARRIER  ▼            │   ← join: wait for all N records
                └───────────────────────────────┘
                            │  one verdict (recorded | escalate)
                            ▼
                      orchestrator (re-decides ONCE per batch:
                                    continue / complete / escalate)
```

- **One FSM state (`workers`).** From the FSM's vantage, exactly one state is active for the whole batch. `String(snapshot.value)` is `"workers"`; `instance.currentState` is `"workers"`; the checkpoint's `machineState` is `"workers"`. The single-active-state spine is untouched.
- **N free-standing children inside the invoke.** The `workers` state's invoke (a `fromPromise` actor, like every other state — `orchestrator.ts:1848`) **creates** N child round-machine actors (`createActor(roundMachine, { input })` + `.start()`) and resolves only when all N have reached a `final` state (`recorded`, `blocked` on escalation §10, or `errored` on a service rejection §6.1/FSM-S5). The children are runtime actor refs the invoke body holds — they are **not** in-context `spawn`-ed regions of the parent's `value` (§6.4), so the join reads each child's final `output` off its own snapshot, not a parent-visible `onDone` (FSM-M2).
- **Barrier-join → one verdict.** When all children reach `final`, the invoke reads their outputs (`Promise.allSettled`, so one child's rejection does not abandon the rest — FSM-S5) and produces a single verdict (`recorded` on the happy path, `escalate` if any lane is `blocked`, `fail`/partial-record on `errored`). The orchestrator is entered **once per batch** and re-decides continue/complete/escalate over a `nodes.json` that grew by N (and a `stop_signals.json` recomputed once at the join, §9).

The orchestrator's existing routing (`workflow.yaml:338-360`) is unchanged in *shape* — it still reads `nodes.json` + `stop_signals.json` and emits one verdict. The only change is that `done_rounds` now advances by N per entry instead of 1.

## 4. Manifest schema changes — how N workers / fan-out is expressed

### 4.1 The `workers` count

Add a `workers` field to `settings` (mirroring `pipeline.parallel.num_workers`):

```yaml
settings:
  mode: docker
  dockerAgent: claude-code
  sharedContainer: true
  workers: 1            # NEW. 1 = today's linear chain (default). 2–4 for production.
  maxRounds: 200        # backstop unchanged (but see §4.4 — must scale with N)
```

`types.ts` `WorkflowSettings` gains `readonly workers?: number` (default 1). `preflight` MAY author it from the experiment (`config.yaml`'s `num_workers`) the same way it authors `sample_n` (search-quality §5.2), writing it into the run spec for the bridge to read — **but** because `workers` governs FSM-level fan-out (not engine sampling), the simplest first cut keeps it a `settings` literal authored at workflow-definition time, not a per-run inferred value. **Recommendation: `settings.workers` literal first; `preflight`-authored later** (§14 OQ1).

### 4.2 Expressing the fan-out state — topology RESOLVED to shape A

The first draft of this doc was self-contradictory (FSM review MF1): §3/§11 asserted the checkpoint's `machineState` is `"workers"` while §4.2 recommended a shape with **no `workers` state**. Both cannot be true, and the shape-without-a-state breaks resume. Resume does:

```
restoredState = String(checkpoint.machineState)          // orchestrator.ts:1558
stateDef      = definition.states[restoredState]         // orchestrator.ts:1559
if (stateDef.type === 'agent' || 'deterministic')        // orchestrator.ts:1560
  replayInvokeForRestoredState(...)                       // orchestrator.ts:1561
```

If `machineState` is `"workers"` but no such state is declared, `definition.states["workers"]` is `undefined` and `:1560` throws on `.type`. If instead `machineState` were `"sample"` (the bare chain head), `replayInvokeForRestoredState` would re-run **`sample`'s** invoke with the *bare, non-lane-templated* `stateDef.run`/`stateDef.resultFile` (`orchestrator.ts:1587-1595`) — one single-`current/` exec, **not** the fan-out. Either way, resume is broken.

**Resolution: shape A — a single real `workers` deterministic-style state whose invoke IS `runFanOutSegment`.** This is the only shape where `machineState:"workers"` is real and resume re-enters the fan-out driver (the task's explicit requirement). The two shapes:

- **(A — CHOSEN) One real `workers` state; its invoke spawns the round child machine.** The manifest declares one `workers` state. Its invoke is `runFanOutSegment` (§6.2), registered in `provideActors` (`orchestrator.ts:1838-1858`) alongside `agentService`/`deterministicService`; its entry/invoke **spawns N instances of the round child machine** (§6.1). The five spoke step-bodies (`sample`/`researcher`/`evaluate`/`analyzer`/`analysis_record` work) become **the child machine's states**, but their **prompts and run-commands stay authored in the manifest** as the declared `workers.segment` the child machine is factored from (`machine-builder.ts`) — shape A does *not* delete the spoke declarations; it requires that the *parent FSM's* active state is `workers`, while the spoke declarations become the child machine's states. On resume, `replayInvokeForRestoredState` re-runs the `workers` invoke = re-runs `runFanOutSegment`, re-spawning children with DB-derived lane reconstruction (§11). `String(snapshot.value)` is `"workers"`, never an object. The diagram in §3 is literal.
- **(B — REJECTED) Keep the spokes as the active states; mark the head with `fanOut`.** Cleaner-looking manifest, but `machineState` would be `"sample"`, and `replayInvokeForRestoredState` would re-run bare `sample`, not the fan-out (MF1). Salvageable only by special-casing the fan-out-segment head inside `replayInvokeForRestoredState` to dispatch to `runFanOutSegment` with lane-templated inputs — a deeper change to the resume codepath than shape A. Rejected for that reason.

**How shape A keeps prompts first-class (the house style) — and why the child machine keeps the per-spoke FSM mechanisms.** The concern that motivated shape B — "FSM carries control flow, skills/prompts carry domain content" (CLAUDE.md "Authoring workflow skills") — is preserved by **declaring the spoke states in the manifest as members of the `workers` segment**, with their prompts/run-commands intact. Under CHANGE 1, the `workers` state factors those declared spokes into a **child machine definition** (§6.1) and creates N free-standing instances of it (`createActor`, FSM-M2). So the spokes are no longer states of the *parent* FSM (the parent is in `workers`), but they are real states of the *child* machine — which means the FSM-owned per-spoke mechanisms (`maxVisits`, visit counts, the `updateContextFromAgentResult` assign) **fire per lane natively from each child's own context**, not "silently no-op." This is strictly better than both the rejected shape B (which kept the spokes as parent states but could not run them concurrently) and the first draft's manual-pump shape A (which kept the spokes declared but bypassed the actor and re-implemented the assigns by hand). The child machine is the right home: parent stays single-active-state in `workers`; the per-spoke machinery runs per lane because each lane is a real actor (§6.1, §7.5). The only mechanism the child machine cannot host is **gate routing** — a child has no parent gate state, which is exactly why a human-gate inside the segment is *unreachable* and an error (§4.3, §10).

Manifest delta (shape A):

### 4.3 Schema + validator rules (`validate.ts`, `types.ts`)

**Prerequisite (SF1) — the new keys must be ADDED to the Zod schemas and `types.ts` interfaces, or they are silently stripped.** The Zod schemas are plain `z.object(...)` with default `.strip()`, not `.strict()`: `workflowSettingsSchema` (`validate.ts:103-117`), `agentStateSchema` (`:54-68`), `deterministicStateSchema` (`:78-87`), `agentTransitionSchema` (`:33-38`, shared by agent + deterministic + gate transitions). So `settings.workers`, `states.workers.fanOut`/`.segment`, and `fanOutMember: true` **parse without error but are dropped** before the definition reaches the pump — which would then read `undefined` for all of them. The implementation contract must name these additions explicitly:
- `WorkflowSettings.workers?: number` (`types.ts:105-157`) + `workflowSettingsSchema` field.
- A `fanOut?: { count: 'workers' | number; join: 'barrier' }` + `segment?: readonly string[]` on the `workers` state's definition (`DeterministicStateDefinition`, `types.ts:267-296`) + `deterministicStateSchema`.
- `fanOutMember?: boolean` on `DeterministicStateDefinition`/`AgentStateDefinition` + their schemas. (`fanOut` lives on the *state*, not a transition, under shape A — so it does not need a home on the shared `agentTransitionSchema`, sidestepping the FSM review's note that `fanOut`-on-a-transition would leak into gate transitions.)

Validator rules (run after the schema additions land):
- **`workers` is a positive integer** (`>= 1`). `0` or negative is a validation error.
- **`fanOut.count` resolves to `settings.workers`** (the literal `workers` keyword, or an inline integer). A `fanOut` state whose `segment` members are not all `fanOutMember: true` is an error.
- **No human-gate state may appear inside the fan-out `segment` — this is an ERROR, not a lint warning (CONSIDER C2).** A lane has no FSM state, so it can never cause `snapshot.matches(gateName)` to fire (`orchestrator.ts:1934-1942`); a gate inside the segment is **unreachable**, not merely discouraged. The validator message states this as an unreachability invariant. The only escalation exit is the segment-boundary drain to `human_escalation` (§10).
- **`workers > 1` with `algorithm: greedy` is rejected (MUST-FIX-2 / §7.4).** Greedy is deterministic in DB state, so concurrent greedy lanes draw identical parents (degenerate). Multi-lane runs must declare a stochastic sampler. (If the run-spec sampler is not visible to `validate.ts` at lint time, this is enforced at run-start in the pump instead, with the same error.)
- **Every `fanOutMember` deterministic state with a `resultFile` writes a lane-scoped path** when `workers > 1` (SF2). The pump is the single source of truth for the lane-templated `resultFile`: it threads the **same** lane path into both the argv and `DeterministicInvokeInput.resultFile`, which `applyResultFile` reads host-side (`orchestrator.ts:2537-2606`, containment checks at `:2552-2564`). A desync between the in-container write path and the host-side `input.resultFile` read silently yields `result file … not found` (`:2566-2568`) → `result_file_error` → `failed` for every lane. The validator checks the argv uses the lane-templated working dir, not the bare `current/`; the Phase 1 unit test asserts the lane-templated path passes the `isSafeWorkspaceRelativePath`/`isWithinDirectory` containment checks (it should — `current/lane_0/…` is workspace-relative).

### 4.4 Linter rules (`lint.ts`)

- **`WF013` (new) — `maxRounds` caps batches, candidate budget is `maxRounds × workers`, AND the round-limit backstop loses lane-internal coverage (§7.5, FSM-S3).** The original WF013 idea warned "`workers > 1` requires `maxRounds` scaled UP because lanes accrue visits N× faster" — that is backwards. Under CHANGE 1, lane visit counts live in each **child** machine's context; the **parent** `isRoundLimitReached` (`guards.ts:21-26`) reads the **parent** `context.visitCounts`, which now increments only on **parent** state entries — i.e. it counts **batches**, not lane-rounds. So `maxRounds` caps batches, and the realized candidate budget is `maxRounds × workers`; if the author intended `maxRounds` as a candidate budget, it must be divided by `workers`. **The harder consequence (a coverage REGRESSION, disclose it): the global `maxRounds` backstop now fires only after `maxRounds` *batches* and is BLIND to lane-internal loops.** In the linear chain, `isRoundLimitReached` over `Math.max(visitCounts)` caught a runaway self-loop on **any** state. Under fan-out it sees only the parent's `workers` count. Per-lane `maxVisits` covers the two states that declare it — `researcher`/`analyzer` (`maxVisits:3`, `workflow.yaml:420`,`:470`) — but `sample`/`evaluate`/`analysis_record` carry **no** `maxVisits`, so a self-loop on one of those inside a lane has **no backstop at all** (the parent guard cannot see lane visits; the child has no round-limit of its own). **WF013 therefore warns on BOTH (i) the parent-batch-vs-candidate budget relationship AND (ii) any self-loopable child state lacking its own `maxVisits`** — such a state needs an explicit per-lane `maxVisits` (or a child-level round-limit guard mirroring `isRoundLimitReached` over the child's `visitCounts`) to restore the backstop the linear chain had. The real per-lane bound is the child machine's own `maxVisits` guard (`isStateVisitLimitReached`, `machine-builder.ts:351-362`, fires natively per child — §7.5), not `maxRounds` — and it only exists where declared.
- **`WF014` (new): fan-out must be barrier-joined.** A `fanOut` with `join` other than `barrier` is rejected (async joins are a non-goal; this lint makes a future async value a deliberate, reviewed addition).
- **Reuse `WF011`** (`lint.ts:430`): a `container: true` fan-out member running before its scope is minted is still caught — but note the scope bundle is now minted **once before fan-out** (§6.2/§8.1), so this lint's invariant is satisfied by construction, not by "first lane mints."

### 4.5 Reconcile the dead parallelism graveyard (CONSIDER-2)

`WorkflowSettings.maxParallelism` already exists and is read nowhere (`types.ts:115`; `validate.ts:109`), alongside the retired `PARALLEL_*` events (`types.ts:435-436`), `parallelResults` (`:458`), and `worktreeBranches` (`:459`). Adding a *new* `workers` field beside a dead `maxParallelism` leaves two parallelism concepts on the type surface. **Decision: delete `maxParallelism` (and the retired `PARALLEL_*`/`parallelResults`/`worktreeBranches` graveyard) in this slice's `types.ts`/`validate.ts` edits**, so there is exactly one parallelism concept (`workers`). The retired-WF005/`parallelKey` lint (`lint.ts:43-45`) is the graveyard marker confirming these are dead, not load-bearing.

## 5. Coordinator concurrency — the one real unknown (audit FIRST)

N sustained concurrent agent tool-call streams (the N `researcher`/`analyzer` lanes) flow through **one** `ToolCallCoordinator` (PolicyEngine, AuditLog, CallCircuitBreaker, ApprovalWhitelist, AutoApprover, ServerContextMap). **This is the single biggest risk** and the **first implementation step** (§14 phase 0). Be honest: until audited, this is "contained feature vs kernel-hardening project."

### 5.1 What is ALREADY safe (verified — de-risks substantially)

The coordinator already holds a **FIFO call mutex** (`AsyncMutex`, fair FIFO, `src/trusted-process/async-mutex.ts:13`; non-reentrant, `:15`; `withLock` at `:42-49`):

- `handleToolCall` runs its whole body — `buildCallToolDeps()` + `handleCallTool()` — inside `this.callMutex.withLock(...)` (`tool-call-coordinator.ts:310-313`).
- `handleStructuredToolCall` likewise (`:348-357`).
- `loadPolicy` acquires call-mutex-then-policy-mutex in a fixed order (`:484-490`), and `close()` does the same (`:588-589`) — no deadlock (documented invariant, `:481-483`).

So the three RMW in-memory caches the mutex exists to protect — **ApprovalWhitelist, CallCircuitBreaker, ServerContextMap** — and the **audit append** (`auditLog.log()` inside `handleCallTool`) are **already serialized** against concurrent lanes. N concurrent `handleToolCall` invocations queue on the FIFO mutex and execute one-at-a-time. **This is the load-bearing safety fact: the coordinator was built for concurrent callers; the mutex is not new work.** Audit entries cannot interleave/tear because the whole pipeline (including the JSONL append) is inside one critical section.

### 5.2 What MUST be audited (the residual unknowns)

1. **Agent-path policy escalation under the held mutex — THE real stall, and drain alone does NOT cure it.** This is the corrected answer to MUST-FIX-4 / FSM review MF3 / devil's-advocate SHOULD-FIX-2. **Verified against the code:** the escalation await *is* inside the held call mutex — `handleCallTool` runs entirely inside `callMutex.withLock` (`tool-call-coordinator.ts:310-313`, `:348-357`), and inside it `handleCallTool` awaits `autoApprove(...)` (`tool-call-pipeline.ts:821`), then `deps.onEscalation(...)` (`:866`) or `waitForEscalationDecision(...)` (`:881`) — all under the lock. So when a `researcher`/`analyzer` lane's tool call hits an `escalate` policy rule (evolve agents are `global` persona with a *real* policy, so this is live, not hypothetical), that lane **holds the FIFO `callMutex` across the entire human-response latency**, and every other lane's tool call queues behind it (`async-mutex.ts:18`, FIFO; non-reentrant). N lanes freeze on one human deliberation.
   - **Why drain-on-escalation does NOT fix this:** §10's drain trips on the **deterministic** `evaluate` verdict `evaluator_blocked` (`workflow.yaml:444-445`), which runs **off-mutex** via `docker.exec` (`orchestrator.ts:2667`) and never touched the call mutex. So drain cures the `evaluator_blocked` case — but that case never stalled the mutex in the first place. The agent-path policy escalation has *no* `evaluator_blocked` verdict to trip the drain; the pump cannot even observe it today (it surfaces inside the coordinator, below the lane's `executeAgentState`).
   - **The honest, minimal answer (Phase 0 decides, but the design commits to this):** the **call mutex must be released across the escalation wait**. The coordinator acquires → evaluates → and *if* the verdict is `escalate`, **releases the mutex, awaits the human (auto-approve + onEscalation/waitForEscalationDecision), then re-acquires to finish the call** (write audit, breaker, transport). This is a small, well-scoped coordinator change confined to `handleCallTool`'s escalation branch (the RMW caches the mutex protects are touched *after* the human decision, so the critical section that needs the lock is the tail, not the wait). It is **in-scope for this slice**, not deferred: without it, `workers > 1` with escalation-prone personas is unsafe (a single escalation latency-cliffs the whole batch). Drain-on-escalation (§10) is retained as the *gate-aggregation* mechanism (one `activeGateId`), **not** as the mutex-stall fix — the two are now cleanly separated.
   - **Fallback if the mutex-release change proves unsafe in Phase 0:** document that `workers > 1` is supported only for personas whose policy *cannot* escalate on the agent path (i.e. the only escalation source is the off-mutex `evaluator_blocked`), and gate N>1 behind that. State this explicitly rather than shipping a silent latency cliff.
2. **AutoApprover concurrency.** The optional LLM auto-approver runs inside the escalation path (`tool-call-pipeline.ts:821`, under the lock today). With the mutex-release fix (item 1), N agent-path escalations *can* be live concurrently → N concurrent auto-approve LLM calls. This is acceptable (each is an independent cheap-model call) but Phase 0 confirms AutoApprover holds no shared mutable state across calls. (Under the rejected drain-only model this was "moot"; with the mutex-release fix it is a real concurrency point to confirm.)
3. **Circuit-breaker false-trip is a correctness risk, made `workers`-aware BY DEFAULT (not optional).** Corrected per CONSIDER-5. The breaker counts identical `(tool, argsHash)` calls in a sliding window (`CallCircuitBreaker`, default 20/60s). With identical-parent lanes (§7.4) all reading the same cognition/candidate files, the calls are **identical, not merely similar** — N lanes × repeated identical reads can trip the window and **deny** legitimate lane work mid-batch, surfacing as a spurious lane failure. This is not "tuning"; it is a default-on correctness bug at `workers > 1`. **Fix: when `workers > 1`, scale the breaker window/threshold by `workers` by default** (e.g. threshold × `workers`), not behind an opt-in config flag. Otherwise gate item 4's "correct breaker counts" passes on a synthetic test while real runs get throttled.
4. **ServerContextMap cross-lane bleed.** The git-working-directory enrichment (`server-context.ts`) is keyed per server, not per lane. N lanes in one container share one git CWD anyway (one `/workspace`), so this is consistent — but confirm no lane-specific context is needed for escalation messages.

### 5.3 Mitigation summary

- **Primary:** rely on the existing FIFO `callMutex` for audit/breaker/whitelist/context-map RMW safety (already shipped, `tool-call-coordinator.ts:310-313`/`:348-357`).
- **Escalation (the real fix):** **release the call mutex across the escalation wait** (item 1) so an agent-path escalation does not freeze every other lane. Drain-on-escalation (§10) is the *gate-aggregation* layer on top (one `activeGateId`), not the stall fix.
- **Breaker:** make the circuit-breaker window/threshold `workers`-aware **by default** when `workers > 1` (item 3) — a correctness fix, not optional tuning.
- **Honesty:** Phase 0 (§14) is "audit the coordinator under N concurrent streams, write the concurrency test (gate item 4), AND land the mutex-release escalation fix." Only after it passes do later phases land. If the mutex-release change surfaces a non-reentrancy bug the FIFO mutex does not cover, the slice either lands the kernel fix or **gates N>1 to non-agent-escalating personas** — that branch is explicit, not hidden.

### 5.4 The ORCHESTRATOR'S own shared mutable state — a separate concurrency audit (FSM-S4, Phase 0 sibling)

§5.1–§5.3 audit the **coordinator**. They say nothing about the **orchestrator's** own per-`instance` mutable state, which N concurrent lane executors now touch from concurrent `executeAgentState`/`executeDeterministicState` calls. The orchestrator mutates `instance` state keyed by **`workflowId`, not by lane** — so every such field is a potential cross-lane race the coordinator audit does not cover. This is a **Phase 0 sibling audit**, not a sub-item of §5: enumerate every `instance.*` mutation on the per-state executor path and confirm each is **lane-keyed, idempotent, or serialized** under N concurrent calls. The known surface:

- **`bundlesByScope`** (`orchestrator.ts:654`, set `:926`) — the check-then-act mint race. **Covered** by the mint-once-before-fan-out fix (§6.2/§8.1); confirm no *other* writer.
- **`currentPersonaByBundle`** (`:668`, cyclePolicy short-circuit `:994`, set `:1037`) — N lanes calling `cyclePolicy` on the same bundle. Homogeneous personas make the first writer win and the rest short-circuit (§8.2), but the **read-then-set window** (`:994` read, `:1037` set) is unguarded across `await`s; confirm the homogeneity argument actually serializes it, or guard it.
- **`mintedServersByBundle`** (`:677`, set `:927`, init `:1516`) — written under the same mint path as `bundlesByScope`; confirm it inherits the mint-once protection.
- **`messageLog`** (`:634`, `:1391`) — N concurrent appenders (same single-writer assumption as cognition §9 and observability §6.3 (b)). Needs serialization or a concurrency-safe writer.
- **Audit append** — covered by the coordinator's FIFO mutex (§5.1), but the orchestrator-side `persona`/`laneId` stamping (§8.4) must be lane-correct under concurrency.
- **`instance.tokens.sessionIds` / `outputTokens`** (`:717`, add `:2298`, accumulate `:1318`) — concurrent `add`/`delete` (`:2494`) from N lanes; `Set` mutation is not atomic across `await`s. Confirm every lane registers and de-registers cleanly (ties to the token-accounting correctness gap, §6.3 (a)).

The mint-once fix (§6.2) covers one row; **the rest are unaudited and must pass this checklist before N>1 ships.** Add it as an explicit Phase 0 gate item alongside the coordinator audit (§14).

## 6. The child sub-machine mechanism (THE hard part)

This is the concrete mechanism for "N concurrent sub-machines." **The first draft mis-described the orchestrator as a "manual pump" and concluded lanes had to be plain objects re-implementing the FSM's bookkeeping. That premise is wrong, and the re-implementation was a shadow-FSM smell.** A second revision over-corrected the other way and called the mechanism in-context XState `spawn` — also wrong: in-context `spawn` lives inside an XState entry action, and the `workers` state's body is a `fromPromise` invoke (plain async TS), not transition land. The honest mechanism is **free-standing `createActor(roundMachine, { input })` + `.start()` instances created and joined inside the invoke body** (FSM-M2). Ground truth, verified against the current tree:

- The orchestrator uses XState's **actor runtime**: `createActor(providedMachine)` (`orchestrator.ts:1389`), `actor.start()` (`:1434`), `actor.subscribe(...)` (`:1864`). The state services are real `fromPromise` invokes whose bodies call `executeAgentState`/`executeDeterministicState` (`provideActors`, `.provide({ actors: { agentService, deterministicService } })`, `:1838-1858`).
- XState **auto-transitions** when a `fromPromise` invoke resolves (`xstate.done.actor.<id>` → `onDone`), and fires the `incrementVisitCount` **entry action** on every state entry (`machine-builder.ts:262`, assign at `:604-608`). `maxVisits` is an XState **guard** (`isStateVisitLimitReached`, closed over the `maxVisitsByState` map, `:351-362`, `:406-411`). `updateContextFromAgentResult` is an XState **assign** that writes `totalTokens`/`outputHash`/`previousAgentOutput` back to context (`:489`).
- The **only** "manual" part is the *resume replay*: `replayInvokeForRestoredState` (`:1554-1608`) re-runs one state's invoke by hand because XState's `resolveState` restores *to* a state but does not *enter* it. That seam is small and well-contained — it is not "XState as a mere transition calculator."

So the right mechanism is **a child machine instantiated per lane**, not a hand-pumped object. **The round sub-chain (`sample → researcher → evaluate → analyzer → analysis_record`) is factored into its own XState machine definition; the `workers` invoke body creates N free-standing instances (`createActor(roundMachine, { input })` + `.start()`), each with a per-lane `input`** (parent/seed/`workingDir`). Each child is a real actor running the same definition over the same `agentService`/`deterministicService` actors — **re-provided per `createActor`** because `roundMachine` is a *different* machine (FSM-S4) — lane-scoped via `input`, so **`incrementVisitCount` / `maxVisits` / `updateContextFromAgentResult` fire per lane automatically from the child's own context — zero re-implementation, within a batch.** N children run concurrently over the one shared container; the parent barrier-joins by reading each child's final-state `output` off its own snapshot (NOT a parent-visible `onDone` — these are free-standing actors, not `invoke`d/`spawn`ed children, FSM-M2). **The per-lane context resets on re-spawn after a crash** (§6.1 below, §6.3) — native bookkeeping is a *within-batch* property, not a crash-durable one.

### 6.1 What a lane IS — a free-standing child actor (NOT a plain object, NOT `LaneTraversal`)

A **lane is a free-standing child actor** (`ActorRefFrom<typeof roundMachine>`), held as a local in the invoke body, not a plain bookkeeping object. The `LaneTraversal` interface of the first draft — with its `visitCounts` / `totalTokens` / `outputHashes` fields — **is deleted; those fields are exactly the child machine's own context**, maintained by the same XState actions the parent already uses.

```ts
// The round sub-chain factored as a child machine definition. buildWorkflowMachine
// is refactored to emit this segment machine; agentService/deterministicService are
// RE-PROVIDED per createActor (roundMachine is a different machine — FSM-S4).
const roundMachine = setup({
  actors: { agentService, deterministicService },   // re-provided per createActor, lane-scoped via input
}).createMachine({
  context: ({ input }: { input: LaneInput }) => ({
    laneId: input.laneId,
    workingDir: input.workingDir,                    // current/lane_<laneId>/  (§7)
    parents: input.parents,                          // distinct per lane (§7.4, CHANGE 3)
    samplingSeed: input.samplingSeed,                // base_seed + laneId (§7.4)
    visitCounts: {},                                 // FSM-M1: fresh per createActor → RESETS on resume re-spawn
    totalTokens: 0,                                  // FSM-M1: fresh per createActor → RESETS on resume re-spawn
    previousOutputHash: undefined,                   // per-lane stall detection, native (within a batch)
    // …same WorkflowContext shape the parent uses, scoped to this lane.
  }),
  states: {
    sample: { /* deterministic; entry incrementVisitCount; invoke deterministicService */ },
    researcher: { /* agent; maxVisits guard; updateContextFromAgentResult on done; service reject → `errored` */ },
    evaluate: { /* deterministic; on evaluator_blocked → `blocked` final; service reject → `errored` */ },
    analyzer: { /* agent; maxVisits guard; service reject → `errored` */ },
    analysis_record: { /* deterministic; on recorded → `recorded` final */ },
    recorded: { type: 'final', output: ({ context }) => ({ status: 'recorded', nodeId: context.recordedNodeId, laneId: context.laneId }) },
    blocked:  { type: 'final', output: ({ context }) => ({ status: 'blocked',  reason: context.blockReason, laneId: context.laneId }) },
    errored:  { type: 'final', output: ({ context }) => ({ status: 'errored',  error:  context.laneError,    laneId: context.laneId }) },
  },
});

interface LaneInput {
  readonly laneId: number;
  readonly workingDir: string;
  readonly parents: readonly NodeRef[];   // the lane's distinct sampled parents (CHANGE 3)
  readonly samplingSeed: number;          // base_seed + laneId (§7.4)
}
```

**The third final state `errored` (FSM-S5).** A child `agentService`/`deterministicService` *rejection* — a thrown error, distinct from the `evaluate` `evaluator_blocked` *verdict* — has no defined barrier behavior today. The child's `onError` for each invoke routes to the `errored` final state (capturing the error in context). At the barrier, an `errored` child output maps to an explicit batch verdict (escalate, fail, or partial-record-then-fail; §6.2/§10), instead of a `Promise.all` rejection that would abandon the N−1 in-flight children. This is why the join uses `Promise.allSettled`, not `Promise.all` (§6.2).

Each child runs the **same** five-state chain the linear round runs today, reusing **the same** `agentService`/`deterministicService` (`orchestrator.ts:1838-1858`, re-provided per `createActor`) — the executors are unchanged; only their `input` is lane-templated (lane working dir, per-lane prompt path §7.3, per-lane `resultFile` §7/SF2). Because the child is a real actor, *within a batch*:

- its `incrementVisitCount` entry action increments **the child's** `context.visitCounts` per lane (`machine-builder.ts:262`),
- its `maxVisits` guard reads **the child's** `context.visitCounts` (`isStateVisitLimitReached`, `:351-362`) — so `researcher`/`analyzer` `maxVisits: 3` (`workflow.yaml:420`, `:470`) is enforced **per lane, natively**,
- its `updateContextFromAgentResult` assign writes **the child's** `totalTokens`/`outputHash` (`:489`) — per-lane token budget + stall detection work with **no manual threading**.

**Durability boundary (FSM-M1/S1 — the one place this "native" claim has an asterisk).** Every field above lives in the child actor's runtime context, and the first-cut resume does NOT persist live child actors (§6.3). On resume, each not-yet-recorded lane is re-created fresh via `createActor` with `visitCounts: {}` and `totalTokens: 0`. **So per-lane `maxVisits` is NOT crash-durable**: a lane that burned 2 of 3 `researcher` visits before a crash comes back with a fresh 3-visit budget. Same for the per-lane token accumulator. This is acceptable for node-set integrity (the un-recorded lane re-runs idempotently, §11.2) but means the per-lane *loop bound* is reset, not preserved, across a crash. The *correctness* fix is migrating the checkpoint to `getPersistedSnapshot()` AND switching to real in-context `spawn` (the two do not compose for free — §6.3); the first cut accepts the reset.

**The shadow-FSM re-implementation the first draft proposed (and §7.5 enumerated) is gone.** The child machine does the bookkeeping (within a batch).

### 6.2 The `workers`-state invoke body = create N child actors, start them, await the barrier

The fan-out segment is driven by `runFanOutSegment(workflowId, context)`, the invoke body of the real `workers` state (shape A, §4.2), registered as a `fromPromise` actor alongside `agentService`/`deterministicService` (`orchestrator.ts:1838-1858`). It is a **plain async TS function**, not an XState entry action — so the children it creates are **free-standing `createActor` instances**, not in-context `spawn`-ed children (in-context `spawn` is only callable from inside an action; FSM-M2). Its job is no longer to *pump* lanes — it is to **create N child actors, start them, and await each child reaching a `final` state** (the barrier):

```ts
private async runFanOutSegment(
  workflowId: WorkflowId,
  context: WorkflowContext,
): Promise<FanOutResult> {
  const instance = this.workflows.get(workflowId)!;
  const N = resolveWorkers(instance.definition);            // settings.workers, default 1

  // === PRECONDITION: mint the scope bundle ONCE, BEFORE fan-out (MUST-FIX-2 / FSM MF2). ===
  // ensureBundleForScope is unguarded check-then-act across awaits (orchestrator.ts:838-839,
  // :864) — its own comment (:820-824) says concurrent lazy-mint races are unhandled. N children
  // each hitting their first `container:true` step would each see bundlesByScope.get(scope)
  // === undefined and mint a SEPARATE container, defeating sharedContainer. So we mint here,
  // serially, before any child starts. Every child then BORROWS the already-minted bundle.
  const scope = resolveFanOutScope(instance.definition);    // the single containerScope of the segment
  await this.ensureBundleForScope(instance, scope);         // single await; no concurrent mint possible

  // === CHANGE 3: barrier-side sampling — one db.sample(n=N) partitions N DISTINCT parents,
  // one per lane, over a diversity-maintaining sampler (§7.4). Each lane gets seed base_seed+laneId.
  const laneInputs = await this.sampleLaneInputs(workflowId, scope, N, context);   // §7.4

  // CREATE + START: N FREE-STANDING child actors (NOT in-context spawn — FSM-M2), each running
  // the round machine with its actors RE-PROVIDED (roundMachine is a different machine — FSM-S4)
  // and a per-lane input. Resume creates only the not-yet-recorded children and synthesizes
  // `recorded` outputs for lanes the DB already has (§11/§6.3).
  const children = laneInputs.map((input) => {
    const child = createActor(this.provideRoundActors(roundMachine, workflowId), { input });
    child.start();
    return child;
  });

  // BARRIER: await every child reaching a `final` state. A free-standing actor fires NO
  // parent-visible onDone, so we await each child's OWN snapshot via subscribe-until-done,
  // and join with allSettled (FSM-S5) so a thrown error in one child does not reject the
  // join and abandon the N-1 in-flight peers.
  const settled = await Promise.allSettled(
    children.map(
      (child) =>
        new Promise<LaneOutput>((res, rej) => {
          const sub = child.subscribe((snap) => {
            if (snap.status === 'done') { sub.unsubscribe(); res(snap.output as LaneOutput); }
            else if (snap.status === 'error') { sub.unsubscribe(); rej(snap.error); }
          });
        }),
    ),
  );
  // Map fulfilled (recorded | blocked | errored final outputs) AND rejected (an actor-level
  // throw the `errored` final did not catch) into a uniform LaneOutput[]; a rejected settle
  // becomes { status:'errored', error, laneId } (FSM-S5).
  const outputs = this.collectLaneOutputs(children, settled);

  // A `blocked` (or `errored`) child trips the drain: the parent calls childActor.stop() on
  // still-running peers so XState never enters their next deterministic state (§10.1 — NOT an
  // AbortSignal; the executors take no signal, FSM-M3). An in-flight docker.exec finishes; its
  // record commits idempotently.

  // Serialized post-barrier work (single writer, §9/§10): cognition promotion, stop-signal
  // recompute, escalation aggregation — all read the children's final OUTPUTS, no manual flags.
  await this.joinBatch(workflowId, outputs);

  return this.batchVerdict(outputs);   // { verdict:'recorded' } | { verdict:'escalate', items } | { verdict:'fail', items }
}
```

> **MUST-FIX-2 / FSM MF2 — the mint-before-fan-out precondition is load-bearing.** As an additional belt-and-braces, `ensureBundleForScope` itself SHOULD gain the in-flight-promise guard its `:820-824` comment already prescribes (store the `Promise<bundle>` keyed by scope before the first `await`, return it to concurrent callers). The pre-fan-out single mint is the design-level fix; the in-flight-promise guard hardens the function against any *other* future concurrent caller. Name both in the implementation contract — the §5 coordinator audit covers the *coordinator*, not this orchestrator-side bundle-mint race, which is a distinct concurrency gap.

**There is no `driveLane` and no per-lane `for`-loop bookkeeping.** Each child actor walks its own sub-chain via XState's normal `onDone` auto-transitions *internal to the child* (when the child's own `fromPromise` invoke resolves); its `incrementVisitCount` entry action, `maxVisits` guard, and `updateContextFromAgentResult` assign all fire from the child's own context. The parent does **not** consume the child's `onDone` — a free-standing actor has no parent to fire it at; the parent watches the child's snapshot reach `status === 'done'` (the subscribe-until-done above). Escalation is **a child final state**, not a manual drain flag: a lane hitting `evaluator_blocked` (`evaluate`'s verdict, `workflow.yaml:445`) transitions the child to a `blocked` `final` state whose `output` carries the reason; a thrown service error routes to `errored` (§6.1). The parent reads those at the barrier (§10) — the manual `drain` object and the §10 hand-rolled aggregation are replaced by **reading the children's final outputs**, while keeping drain-on-escalation *semantics* (a blocked child makes the parent `stop()` running peers at their next actor-transition boundary; see §10.1 for the honest abort granularity).

Key properties:

- **Bundle minted once, before any child.** `runFanOutSegment` calls `ensureBundleForScope` *before* creating any child, so the scope bundle exists when the first child runs and every child reuses the same `containerId` (`bundlesByScope.get(scope)` hits, `orchestrator.ts:838-839`). No N-container race.
- **N concurrent `docker.exec`s into one container.** Each child's deterministic states drive `deterministicService` → `bundle.docker.exec(bundle.containerId, …)` (`orchestrator.ts:2667`). N children call it concurrently against the **same `containerId`** — exactly the model (§8). The serial `for` in `reduceDeterministicCommands` (`:2615-2641`, the loop body at `:2622`) is *intra-state* (one command array per spoke); the fan-out concurrency is *across child actors*, so that loop is untouched (FSM review C4 confirms this read). Note the abort consequence (FSM-M3, §10.1): `childActor.stop()` can prevent the child *entering* its next deterministic state, but it cannot interrupt a queued-but-not-started exec inside one multi-command `reduceDeterministicCommands` call, nor a mid-exec one — those take no `AbortSignal`.
- **The parent FSM sees one invoke.** From the parent machine's vantage, the `design` edge invokes the `workers` state once; `runFanOutSegment`'s promise resolves once (the barrier); the parent actor receives one `xstate.done.actor` event and transitions back to `orchestrator`. **`String(snapshot.value)` is `"workers"`, never an object** (`:1873`) — the children are free-standing runtime actor refs the invoke body holds, and their snapshots are *not* folded into the parent's `value`. `instance.currentState` is `"workers"` for the whole batch. (This is the precise distinction §6.4 now draws: free-standing child ACTORS preserve single-active-state; parallel REGIONS would not.)
- **Executors unchanged; the ASSIGN half is the child's, not re-implemented.** `executeAgentState`/`executeDeterministicState` run unchanged except for the lane-templated `input` (lane working dir, per-lane prompt §7.3, per-lane `resultFile` §7/SF2). The actor's entry/`updateContextFromAgentResult` assigns that the first draft proposed to re-do by hand are simply **the child machine's own actions** — the FSM review's "you dropped the assign half" objection is dissolved by giving each lane a real actor that keeps both halves (within a batch; reset on resume, §6.1).

### 6.3 How a lane's progress is tracked — DB-as-truth + re-spawn; `batch.json` DELETED (SF3 resolved)

The FSM checkpoint stays flat (`machineState: "workers"`). The question is how resume reconstructs which lanes already recorded. The first draft proposed a `current/batch.json` ledger; the FSM review (SF3) demoted it to advisory. **CHANGE 1 deletes it outright.** The first-cut resume rests on **one** durable fact — the engine DB — and explicitly does NOT depend on XState child persistence:

1. **The engine DB is the SOLE authority for "did this lane record."** The only durable, trustworthy fact is "is this lane's `step_name` already a recorded node?" — answered directly from `nodes.json` via `_find_recorded_node_for_step` (`evolve_result.py:186-191`). With lane-tagged step names `step_<batch_index>_lane_<k>` (§7.2), `runFanOutSegment` reconstructs the batch entirely from the DB:

   ```
   for k in 0..N-1:
       if nodes.json contains a node with meta_info.step_name == f"step_{batch_index:04d}_lane_{k}":
           lane[k] is RECORDED   # do not create a child; synthesize its `recorded` output
       else:
           createActor + start a fresh child for lane k    # re-run from `sample`
   ```

2. **In-flight child state is NOT persisted in the first cut — and CANNOT be without a migration (FSM-M2).** This is the resolved fallback: the children are **free-standing `createActor` instances local to the invoke body**, not in-context `spawn`-ed children — so they are **not** captured by the parent's `getPersistedSnapshot()` *even if we switched to it*. (XState v5 only serializes children created via in-context `spawn`/`invoke`, which lives in the parent's actor system; a free-standing `createActor` is its own root.) Combined with the fact that the current checkpoint is `{ machineState, context }` (not a persisted snapshot at all, below), **DB-as-truth + re-spawn is the only viable first-cut resume.** No `batch.json` exists in any form. The checkpoint is the FSM checkpoint plus the DB.

> **PERSISTENCE CAVEAT — flag for reviewers (the one place CHANGE 1's premise needs pressure-testing, now resolved to DB-as-truth).** The codebase does **not** currently use `getPersistedSnapshot()`. It checkpoints `{ machineState: snapshot.value, context: snapshot.context }` (`buildCheckpoint`, `orchestrator.ts:1977-1999`) and restores via `resolveState({ value, context })` + `createActor(machine, { snapshot })` (`:1457-1461`). A `value + context` round-trip does **not** rehydrate live child `ActorRef`s — those are runtime objects. Making in-flight children crash-durable is **not** a checkpoint-format swap alone; it requires BOTH (a) migrating the checkpoint to `actor.getPersistedSnapshot()` AND restoring via `createActor(machine, { snapshot: persisted })`, AND (b) switching from free-standing `createActor` children to in-context `spawn` so the children are part of the parent's serialized actor system in the first place. **(a) and (b) do not compose for free** — each is a real change to the resume codepath, and (a) without (b) persists a parent whose children are simply absent. **The first cut takes neither**: DB-as-truth + re-spawn — never persist a mid-batch child; on resume, re-create every not-yet-recorded lane from `sample`, synthesizing `recorded` outputs for lanes the DB already has. It costs only re-running un-recorded lanes (idempotent, §11.2) and the per-lane `maxVisits`/token reset (§6.1, FSM-M1). The `getPersistedSnapshot()` + `spawn` migration is the *correctness* fix for per-lane durability (it is what makes a re-spawned lane keep its visit budget), flagged as the clean long-term answer, not assumed done.

**`batch_index` is "1 + max recorded batch_index," NOT `floor(done_rounds / N)`** (CONSIDER-4 / FSM-C1). After a *drained* batch (§10) only some lanes record, so `done_rounds` advances by less than N, and `floor(done_rounds / N)` would collide or skip — breaking the `step_<batch_index>_lane_<k>` uniqueness the idempotency story rests on. `batch_index` is therefore an independent monotonic counter, **recoverable from the DB as `1 + max(batch_index seen in any recorded step_name)`**. It is monotonic only across *recorded* batches: a batch that drained with **zero** records leaves no `step_<batch_index>_lane_<k>` node, so on resume `1 + max(...)` yields the **same** `batch_index` again and the drained batch reuses its index — **intended and safe**, because step-name uniqueness only requires that no two *recorded* nodes share a `step_<batch_index>_lane_<k>`, and a re-run of a fully-drained batch records under the index it never used. Derivation from a node count that advances non-uniformly is explicitly rejected.

**Observability fan-in is the named residual cost — and one of its four facets is CORRECTNESS, not cosmetic (CONSIDER C1 / FSM-S2).** This is what the child machine does *not* give for free. The parent's `subscribeToActor` is built around **one** actor: a single `String(snapshot.value)` (`:1873`), a single `instance.currentState`, a flat `transitionHistory` (`:1890-1897`), one `activeGateId`. The children's own transitions are not folded into the parent's snapshot. The full surface that must be fanned in:

- **(a) Live token accounting — CORRECTNESS, flagged.** Token *accumulation* itself survives fan-out: `setupTokenSubscription` (`orchestrator.ts:1312`) subscribes to the **token-stream bus** (`getTokenStreamBus().subscribeAll`) keyed on `instance.tokens.sessionIds` (`:1317`), which is state-independent, and each lane's `executeAgentState` registers its session via `sessionIds.add(agentSessionId)` (`:2298`) — so N lanes' `message_end` events all land in `instance.tokens.outputTokens` regardless of the parent being frozen at `workers`. **But two real gaps remain.** (i) The session-id registration MUST happen for every lane child's agent invoke (it does, via the same `executeAgentState`), and must be confirmed under concurrent lanes — an unregistered lane silently drops its tokens (Phase 0 / §5.4 audit item). (ii) The **per-lane `totalTokens` write-back to context** (`updateContextFromAgentResult`, the parent path at `:2452` writes `instance.tokens.outputTokens` into context `totalTokens`) is the child's context now, and the **enforcement** StopCondition runs on **parent** step transitions — and the parent makes **one** transition per batch (`orchestrator → workers → orchestrator`). So the budget *guard* (`BudgetExhaustedError`) only re-evaluates **once per batch, at the barrier**, not between lane steps: a fan-out batch can overshoot the budget by up to one batch's worth of N-lane spend before the guard fires. With N multiplying the burn rate, the overshoot window is N× wider than the linear chain's per-step granularity. The mitigation is to surface per-lane token deltas to the budget guard *during* the batch (forward each child's `message_end`/`updateContextFromAgentResult` delta to an in-batch budget check), or to accept and document the one-batch overshoot. This is enforcement granularity, not pure observability — call it out, do not defer as cosmetic.
- **(b) `messageLog` concurrent writers.** N children appending to the one `MessageLog` (`:634`/`:1391`) is the same single-writer assumption that bites cognition (§9). Either serialize the appends or give the log a concurrency-safe writer.
- **(c) The lifecycle event bus.** `emitLifecycleEvent` fires **one** `state_entered: workers` per batch — downstream consumers (web UI, daemon) see a single opaque `workers` span, not N lane sub-spans, unless children emit upward.
- **(d) `getDetail`'s `visitCounts`.** `getDetail` returns the **parent** (batch) `visitCounts`, not per-lane — an operator inspecting a running batch sees batch counts, not which lane is on its third `researcher` visit.

Per-lane transition history requires **subscribing to the child actors** (each is a real actor with its own `.subscribe`) or having children emit upward (XState `sendParent` / an `emit`-to-parent channel). **This is principled XState work — fan-in of child-actor observability — NOT shadow-FSM re-implementation** — but (a) is load-bearing for budget enforcement and must land with the N>1 path, not be deferred as cosmetic. Recommended mitigation: subscribe to each child and (i) forward token deltas to the budget tracker, (ii) append `laneId`-tagged **message-log** entries (freer-form than `TransitionRecord`, need not stay single-valued). A `laneId` on `TransitionRecord` itself is *not* added — that record is the parent FSM's single-active-state ledger by contract.

### 6.4 Free-standing child ACTORS vs parallel REGIONS — the distinction the first draft conflated

The first draft rejected "XState parallelism" wholesale. **That over-rejected.** There are two different ways to get concurrency under XState, with opposite consequences for the single-active-state spine, and only one is forbidden:

- **Parallel REGIONS (`type: 'parallel'`) — correctly REJECTED.** A `type:'parallel'` machine's snapshot value is an **object** → `String(value)` → `"[object Object]"` (`orchestrator.ts:1873`), which breaks checkpoint serialization (`machineState`), terminal detection (`isTerminalStateValue`, `:1970`), and gate matching (`snapshot.matches`, `:1934`). Parallel regions live *inside* the parent's `value`, so they make `value` an object — fatal to the spine. This is the thing the doc must keep rejecting.
- **Free-standing child ACTORS (`createActor` + `.start()` inside the invoke body) — the RIGHT tool, and the one this slice uses.** A free-standing child actor is its **own root actor**, held as a local in `runFanOutSegment`, not a region of the parent's `value` and not a child of the parent's actor system. The parent's `snapshot.value` stays the string `"workers"` (`:1873` untouched); the children's snapshots are *their own*, read by the parent via `subscribe`. So this preserves the single-active-state spine exactly as well as the (now-deleted) plain-object pump did — but with the child's FSM machinery (visit counts, `maxVisits`, context assigns) running natively per lane (within a batch, §6.1). The trade-off (FSM-M2/§6.3): being its own root means a free-standing child is **not** in the parent's `getPersistedSnapshot()`, which is *why* resume is DB-as-truth + re-spawn.

The earlier blanket claim "`spawn`/`sendTo` are unused by the machine builder, avoid them" described the *machine builder's static state emission* (it emits 4 flat state kinds, `machine-builder.ts:357-377`) — true and unchanged for the **parent** machine. The **child round machine** is a separate definition the `workers` invoke body instantiates at runtime via `createActor`; doing so does not make the parent's `value` an object and does not touch the builder's flat-state emission. The guardrail is therefore narrowed and made precise: **no `type:'parallel'` (it breaks `String(snapshot.value)`); free-standing `createActor` child actors are allowed and are how the fan-out works.** (In-context `spawn` — a child *of the parent's* actor system — is a *third* option, distinct from both: it would be captured by `getPersistedSnapshot()` and is the path to crash-durable in-flight children, but it is NOT used in the first cut, §6.3.) The concurrency is N real child actors, joined at the barrier; the parent stays single-active-state.

## 7. Lane discipline — distinct working areas, no collision

The single-`current/` design (`evolve_result.py:54-55`) and count-derived `step_name` (`:575-576`) collide under N lanes: all N would write `current/step_name`, `current/context.json`, `current/result.json`, and all N would compute the **same** `step_name` from the same `done_rounds` at fan-out time. Resolution:

### 7.1 Per-lane working sub-paths

Each lane gets its own working area: **`current/lane_<laneId>/`** instead of `current/`. The bridge's `_current_dir(run_dir)` (`:54-55`) gains a lane parameter (a `--lane <k>` flag on every bridge subcommand, or a `EVOLVE_LANE` env var the pump sets per `docker.exec`):

```python
def _current_dir(run_dir: Path, lane: int | None = None) -> Path:
    base = run_dir / "current"
    return base / f"lane_{lane}" if lane is not None else base
```

The **lane-scoped** scratch files (`step_name`, `context.json`, `sample.json`, `result.json`, `analysis.md`, `analysis_record.json`, `cognition_item.json`) move under `current/lane_<k>/`. The manifest's `run:` argv for each `fanOutMember` state is lane-templated by the pump (`laneTemplated`, §6.2): the `--context-file`/`--result-file` paths become `…/current/lane_<k>/…`. `workers: 1` keeps the bare `current/` (no `lane_` sub-path) for byte-identical backward compat (gate item 1).

**`stop_signals.json` is the one exception — it is shared/barrier-owned, NOT lane-scoped** (SHOULD-FIX-1). The first draft said "*all* per-round scratch files … move under `current/lane_<k>/`," but `_clear_current_round` (`evolve_result.py:62-72`) also lists `stop_signals.json`, which §9 recomputes **once** at the join into the bare `current/stop_signals.json` (`evolve_result.py:328`, `:747`, `:800`) — the single file the orchestrator prompt reads per batch. So the file split is:

| File | Scope | Owner |
| --- | --- | --- |
| `step_name`, `context.json`, `sample.json`, `result.json`, `analysis.md`, `analysis_record.json`, `cognition_item.json` | **lane-scoped** (`current/lane_<k>/`) | the lane |
| `stop_signals.json` | **shared** (bare `current/`) | the barrier join (§9), written once |

`_clear_current_round` is correspondingly split: the per-lane clear (at each lane's `sample`) wipes only that lane's sub-dir; `stop_signals.json` is **not** touched by a per-lane clear (a per-lane clear of a shared file would race or wrongly wipe the batch signal). The shared `stop_signals.json` is owned by the join's `recomputeStopSignals` (§9), which overwrites it once per batch.

### 7.2 Distinct step names without collision

`step_name = f"step_{done_rounds + 1:04d}"` (`:575-576`) computed from a shared `done_rounds` would give all N lanes the same name. Two options:

- **(a) Offset by laneId at sample time:** `step_name = f"step_{done_rounds + 1 + laneId:04d}"`. Cheap, deterministic, but assumes `done_rounds` is stable across the batch (it is — no lane records until the barrier-adjacent record step, and the record step is where ids are assigned). **Risk:** if lanes record at slightly different engine `next_id`s, the *step_name* (cosmetic, used for idempotency keying and step dir) is offset-derived while the *node id* is engine-assigned-atomically — these can diverge, which is fine (step_name is a label, node id is the key) but must be consistent within a lane.
- **(b) Lane-tagged step names:** `step_name = f"step_{batch_index:04d}_lane_{laneId}"`. Unambiguous, never collides, and makes the idempotency key (`meta_info.step_name`, correctness §6.2) trivially lane-unique. The step dir is `steps/step_<batch>_lane_<k>/`.

**Recommendation: (b) lane-tagged step names.** The correctness slice's idempotency guard keys on `meta_info.step_name` (`_find_recorded_node_for_step`, `:186-191`); a lane-tagged name makes each lane's record independently idempotent (gate item 6) with zero ambiguity, and the engine assigns node ids atomically on `record` regardless of step_name (§8). Option (a)'s offset arithmetic is fragile under resume (a half-recorded batch changes `done_rounds`); (b) is resume-stable because `batch_index` (the monotonic counter, §6.3) and `laneId` are both durable.

### 7.3 Agent prompts hardcode `current/` — they MUST be lane-templated too (MUST-FIX-3)

This is a real gap the first draft missed. §6.2 lane-templates the *argv* of deterministic states (`laneTemplated(stateDef.run, lane)`), but the `researcher` and `analyzer` **agent prompts** embed literal `current/` paths in free text, which no argv-templating reaches. The `analyzer` prompt (`workflow.yaml:457-462`) directs the agent at three hardcoded paths: read `/workspace/.evolve_runs/main/current/context.json`, read `…/current/result.json`, write `…/current/analysis.md`. The `researcher` prompt similarly points at `current/` context. A lane running the unmodified prompt reads/writes the **shared** `current/`, so two lanes' `analyzer` agents both overwrite `current/analysis.md` — exactly the cross-lane clobber §7.1 claims to eliminate, but on the *agent* path, which §7.1 alone does not address. **The claim "agent prompts unchanged / reuses executors verbatim" was false and is retracted.**

**Resolution: the pump substitutes a per-lane path into the prompt before `executeAgentState`.** Two mechanisms; pick (a):

- **(a — CHOSEN) Prompt path templating.** Rewrite the manifest prompts to use a `{laneDir}` placeholder (e.g. `Read {laneDir}/context.json …`). The substitution is driven by the **child machine's per-lane `input`**: the child's `researcher`/`analyzer` invoke passes the lane-templated prompt into `executeAgentState` (via the `LaneInput.workingDir` → `{laneDir}` → `/workspace/.evolve_runs/main/current/lane_<k>`, or the bare `current` at `workers: 1`). This **does edit the manifest prompts** — which contradicts the original (shape-B) rationale for keeping prompts "unchanged"; under shape A the prompts are still authored in the manifest (house style preserved) but now carry a `{laneDir}` template the child fills from its `input`. The substitution is a pure string replace; the domain content of the prompt is untouched.
- **(b — alternative) Lane-dir via env/context file.** Inject the lane dir as an env var or a small context file the agent is told to read, and rewrite the prompts to use a relative `current/` resolved against a lane-scoped working directory the session sets. More moving parts (the agent must be told to consult the env/file), and the session's working-directory plumbing is less direct than a prompt substitution. Noted, not chosen.

Either way the gate is the same: **a `workers: 3` run must show each lane's `analyzer` reading and writing its own `current/lane_<k>/analysis.md`, with no shared-`current/` agent writes.** Add this assertion to the Phase 4 integration test (§14).

### 7.4 Lane divergence is INJECTED at `sample` — a first-class barrier-side `sample(n=N)` (CHANGE 3a)

§7.1–7.3 make *working dirs*, *step names*, and *prompt paths* distinct — **not the sampled parents.** Distinct working dirs with identical parents is the degenerate case: with the default greedy sampler, `GreedySampler.sample` is `sorted(nodes, key=score, reverse=True)[:n]` (`evolve_core/algorithms/greedy.py:13-17`), a pure function of DB state; `db.sample` reads under `_database_guard` but **does not mutate** (`database.py:53-54`, `:206-215`); no lane records until the barrier-adjacent `analysis_record`. So N independent per-lane `sample` calls reload the **same** `nodes.json` and draw the **same** top-N parents → near-duplicate candidates ("N samples of one distribution," not an N-wide frontier). This is rejected (§4.3 validator rule: `workers > 1` + `algorithm: greedy` is an error).

**Divergence is therefore a first-class mechanism, injected at the chain head `sample`, not a temperature accident.** It is realized as **one barrier-side `db.sample(n = N)` over a diversity-maintaining sampler** (island / MAP-Elites — `IslandSampler` already exists, `evolve_core/algorithms/island.py`, MAP-Elites within one DB), executed in `runFanOutSegment` *before* spawning (the `sampleLaneInputs` call, §6.2). It hands each lane a **distinct** parent partition, with per-lane sampler seed `base_seed + laneId` reusing the search-quality slice's seeded-sampler machinery (`sampling_seed.txt`, `_run_seeded_helper`, `evolve_result.py:367-368`, `:397`) so the partition is **reproducible**, not timing-dependent. Each lane's `sample` then writes its **own** `current/lane_<k>/context.json` from its assigned parents, and divergence flows `sample → researcher → evaluate` for free (no further per-step coordination). An optional per-lane strategy hint (a different exploration directive per lane) can ride the same `LaneInput`.

This **resolves** the first draft's open choice between "per-lane seed offset" (independent draws) and "barrier-side partition." The barrier-side `sample(n=N)` is now the recommended mechanism (it guarantees disjoint parent sets even on a base that would otherwise collide, and centralizes the one DB-read into a single serialized call at the boundary — cheap, since `sample` is a read), with the per-lane seed offset folded in *inside* it (seed `base_seed + laneId` per partition). The cost — one serialized `db.sample` at fan-out — is small (a single guarded read, `database.py:206-215`), and it is the natural place to enforce diversity.

**Distinct inputs ≠ distinct outputs — content-hash candidates at fan-in (the honest guard).** Distinct parents do **not** guarantee distinct candidates: the researcher LLM can still collapse two different parents into near-identical edits, so "effective N" can be below N. Rather than *assume* the divergence worked, **content-hash each lane's recorded candidate at the barrier and log the per-batch duplicate rate.** Reuse the cognition store's existing content-hash trick (`hashlib.sha256(...).hexdigest()` over the lesson, `evolve_result.py:419`, with duplicate detection at `:433`) — apply the same `sha256` over each lane's candidate at fan-in, count collisions, and surface the duplicate rate as a per-batch diagnostic. **This surfaces wasted lanes instead of pretending effective-N = N.** It is observability, not a hard gate (duplicates are recorded as distinct nodes; the metric tells the operator the sampler/diversity knobs need tuning).

### 7.5 Lane-state bookkeeping is the child machine's own context — no manual threading (CHANGE 1 supersedes the shadow-FSM table)

**The first draft's §7.5 was a four-row table of FSM mechanisms the "manual pump" had to re-implement by hand. The "you kept execute and dropped assign" objection that motivated the table is dissolved: the child keeps *both* halves.

The parent machine keeps its flat states; the segment becomes a separate machine the `workers` invoke body instantiates. This is the load-bearing implementation task of Phase 3.

## 8. Container, policy, and DB concurrency

### 8.1 Shared container, N concurrent execs — bundle minted ONCE before fan-out

`evolve` is `sharedContainer: true` (`workflow.yaml:8`). All N lanes exec into the **one** bundle for the (single) `containerScope`. **Critically, that bundle is minted once by `runFanOutSegment` BEFORE any child is created (§6.2), not lazily by "the first lane."** The original "first lane mints, rest reuse" framing was unsafe: `ensureBundleForScope` is unguarded check-then-act across `await` points (`orchestrator.ts:838-839`, `:864`), and its own comment (`:820-824`) states concurrent lazy-mint races are unhandled — N lanes racing their first `container: true` step would each see `bundlesByScope.get(scope) === undefined` and mint a **separate** container, defeating `sharedContainer`. With the pre-fan-out single mint, the bundle exists before any lane runs; every lane's first `container: true` step hits `bundlesByScope.get(scope)` (`:838-839`) and reuses it. The N lanes are then N concurrent `bundle.docker.exec(bundle.containerId, …)` (`:2667`) into the same `containerId`, sharing `/workspace` (the host mount), so the engine DB at `.evolve_runs/main/database_data/` is shared. (Belt-and-braces: also add the in-flight-promise guard to `ensureBundleForScope` itself, §6.2.)

**Resource-model caveat — `:2667` is exactly where the off-mutex win can evaporate (§0.3).** "N concurrent execs into one container" overlaps only when the evals do not contend on a single shared scarce resource. For CPU-light/memory-bounded evals (the validation target, §14 OQ6) they overlap up to the container's core/RAM budget. For the GPU/TPU/memory-bound flagship domains (matmul kernels, FlashAttention tiling, TPU circuits) they **do not** — N evals serialize on one accelerator or OOM each other (`/tmp/alphaevolve.txt:2180`). The resource model that fixes this (per-lane cgroup/GPU isolation, N containers, or a host-side bounded eval queue) slots in **at this exec point** (`:2667`) without touching the FSM topology or the lane machinery (§6); it is the bridge to distributed evaluator scheduling and is out of scope for this sync slice. Cap N to the container's core/RAM budget (or the device count) until that resource model lands.

### 8.2 Single policy swap (homogeneity)

All lanes are `global` persona. `cyclePolicy` (`:988`) short-circuits when the bundle already has the persona loaded (`currentPersonaByBundle.get(bundle.bundleId) === persona`, `:994`). So the policy is loaded **once** (by the first lane's first agent state) and the other N-1 lanes hit the short-circuit — **no swap race**. The audit `persona` field (`audit.ts:46`) is single-valued and correct because there is only one persona. **This is why homogeneity is load-bearing:** a heterogeneous batch would need N policies on one coordinator concurrently, which the single `policyEngine` field (`tool-call-coordinator.ts:531`) cannot represent. Lane attribution comes from a **new lane-id**, not a persona change (§8.4).

### 8.3 The engine DB is already cross-process safe

Verified: every DB mutation goes through `_database_guard` (`database.py:207-215`), which takes the in-process `RLock` (`:208`) **and** the cross-process `InterProcessFileLock(self.lock_path)` (`:209`), reloads-from-disk inside the lock (`refresh=True` → `_load_locked`, `:210-211`), and saves atomically via `os.replace` of a temp file (`_save_locked`, `:162`). So concurrent lane `sample`s and `record`s from N separate `docker exec` Python processes are **already safe**: each sees a consistent reloaded view and writes atomically. `add_with_previous_nodes` assigns ids under this guard (`:67-74`). **No engine change.** (The test hook `EVOLVE_DB_TEST_HOLD_LOCK_MS`, `:212`, lets the concurrency test force lock contention deterministically — useful for gate item 2.)

### 8.4 Lane-id on audit entries

Add an optional `laneId?: number` to `AuditEntry` (`src/types/audit.ts:3-47`) for attribution. Because the persona is homogeneous, the lane-id is the only thing distinguishing N concurrent streams in the audit log. The coordinator stamps it the same way it stamps `persona` (from a per-call context the pump threads through `executeAgentState`'s `laneId`, §6.2). This is a **non-policy** attribution field — it changes nothing about evaluation, only audit readability. (Alternatively, encode the lane-id into the `requestId` prefix to avoid a schema field; the explicit field is cleaner for querying.)

### 8.5 The scheduler-factory seam — the path to distributed / resource-isolated eval (CHANGE 2)

§0.3 said the GPU/TPU flagship domains need per-lane resource isolation, a fleet of evaluator nodes, or a bounded eval queue, and that this is the bridge to distributed evaluator scheduling. **That bridge is realized through a seam that already exists — the resource model is a different infrastructure-factory implementation, not a rewrite of the FSM.**

**The placement seam already exists.** The orchestrator never creates a container directly. It calls the injected `createWorkflowInfrastructure` factory (deps field `orchestrator.ts:463`, called at the bundle-mint boundary `:862`; the default localhost implementation is loaded lazily by `loadDefaultInfrastructureFactory`, `:1216`, which wraps `createDockerInfrastructure`). After minting, the orchestrator only ever touches the bundle through `bundle.docker.exec(bundle.containerId, …)` (`:2667`). **The FSM talks to bundles, not nodes.** So:

- A **localhost emulation scheduler** is the default factory (the current `createDockerInfrastructure`), placing every eval-slot on the same host. This is option 0 of §0.3 — and it is what this slice ships.
- A **resource-isolated scheduler** places eval-slots with per-slot resource budgets. **Split the isolation claim honestly (DA-S1):** **CPU/mem caps are real and available per-container today** — `--cpus` / `--memory` are standard Docker run flags, so a fat-memory lane can be capped so it cannot OOM its peers, on localhost, now. **GPU isolation is a harder, host/driver-dependent follow-up, NOT in the same bucket.** MIG slicing is datacenter-GPU-only (A100/H100-class) and **statically pre-partitioned** (you cannot carve MIG slices on demand per lane); time-slicing a GPU across containers gives **no memory isolation** and therefore does **NOT** fix the `/tmp/alphaevolve.txt:2180` OOM (N processes time-slicing one GPU still share its VRAM and still OOM). So a "resource-isolated scheduler" delivers CPU/mem isolation cheaply but GPU isolation only on specific hardware with static partitioning — list them separately, not as equally-available knobs.
- A **distributed scheduler** places eval-slots on *other hosts* (§0.3 option 3, the evaluator fleet). Same `docker.exec` contract; the slot's `docker` handle targets a remote daemon.

**The pool is a NEW lifecycle the factory does not provide (DA-S3 — the real factory strain).** `createWorkflowInfrastructure` is **create-one / destroy-one**, with the mint-once-per-scope-reused-till-terminal discipline (`ensureBundleForScope`, `:834`; teardown `:1072`). A **pool** is a *different* lifecycle: **acquire / release**, a **cap to the device budget**, and a **queue** for the overflow. The factory does not express any of acquire-release, capacity, or queueing — so the eval-pool (Phase 8) is a **new lifecycle abstraction layered over the factory**, not a config of it. Two implementations to decide between: **per-eval-create** (a fresh eval-slot per `evaluate` call → simple but container-churn-heavy) vs **pooled-reuse** (a fixed set of eval-slots sized to the device budget, acquired and released per call → leaning this way, since the slots are thin and reuse avoids churn). The factory creates the *infrastructure*; the pool owns the *lifecycle*. State this so the scheduler design does not assume the factory already pools.

**The transport seam is about AGENT-bundle mediation — IRRELEVANT to the eval-slot (DA-C2).** The proxies' **TCP mode** (`docker-infrastructure.ts:454`, `listenMode: useTcp ? 'tcp' : 'uds'`, socat sidecar `:290-293`, `:1029-1077`) and the **coordinator control socket** (UDS `controlSocketPath`, `orchestrator.ts:848`; loopback-only TCP bind `control-server.ts:341`) exist to mediate an **agent** bundle's MCP/MITM traffic and policy hot-swap. **The eval-slot has none of that** — it is coordinator-less (DA-M1, §8.6), so the control transport and proxy plumbing simply do not apply to it. **The eval-slot's cross-host blocker is NOT the control transport — it is the shared `/workspace`/DB mount.** Same-host placement keeps the file-locked engine DB (`database.py:207-215`, already cross-process safe, §8.3) shared **for free**; cross-host placement breaks exactly one thing for the eval-slot: the DB on a *local* FS per host must become a networked/shared store. (The agent bundles' control-transport-network-addressability is a *separate* cross-host concern that belongs to the agent half, not the eval-slot half.) **Same-host emulation is the right first step** because it defers the DB-sharing problem entirely; the factory/pool seam makes the cross-host follow-up additive.

### 8.6 Two-pool scheduling — only `evaluate` is the schedulable unit (CHANGE 3b)

The AlphaEvolve pipeline is **two pools**: LLM samplers and evaluation nodes, behind one controller (`/tmp/alphaevolve.txt:459`; the per-candidate loop `sample → generate-diff → evaluate → add` is `:189-196`). This slice mirrors that split because the two kinds of per-lane work have opposite resource profiles. **Note the routing rationale precisely (DA-C3):** `sample` and `analysis_record` are **also** deterministic `bundle.docker.exec` calls — so "deterministic exec" is NOT what distinguishes `evaluate`. `evaluate` is routed to the pool by **resource profile**: it is the 100-compute-hour unit (§0), the heavy one. `sample`/`analysis_record` are cheap DB touches that stay in the shared container. The split is by cost, not by step kind.

- **Agent steps (`researcher`, `analyzer`) are LLM API egress.** The heavy compute (LLM generation) runs **remotely** at the provider; locally these are lightweight (think-then-write-one-diff, `config.yaml:44-49`, plus a small tool-call footprint). They **stay in the shared agent container** — co-locating N agents is cheap because the local cost is just API round-trips through the coordinator.
- **`evaluate` is the heavy local compute.** For the target domains the per-candidate eval is the 100-compute-hour unit (§0). It is the **schedulable unit** → dispatched to a **resource-budgeted eval-slot pool**, *not* run as an unbudgeted `docker.exec` into the shared container (`orchestrator.ts:2667`).

**The eval-slot is a THIN trusted-exec target, NOT a `DockerInfrastructure` bundle (DA-M1/M2 — the most load-bearing two-pool fix).** This corrects the prior framing that called the eval-slot "the same bundle minus some parts" or "a different factory impl returning a `DockerInfrastructure`." It is **a narrower object than a bundle.** Here is why, grounded in the code: `evaluate` is `type: deterministic, container: true`, run as a **trusted** `bundle.docker.exec(containerId, YAML-argv)` (`orchestrator.ts:2667`). `src/trusted-process/CLAUDE.md` (`:17-20`) states this is **intentionally NOT a coordinator tool call** — "the command argv comes from trusted workflow YAML, not from the [agent]." So the eval path touches **none** of the agent-mediation machinery a `DockerInfrastructure` bundle carries: **no coordinator, no policy engine, no MITM, no MCP relays, no CA, no fake sentinel keys, no control server.** All of that exists to mediate *agent-authored* tool calls; the eval-slot runs *trusted YAML argv* and needs none of it. The eval-slot is therefore just:

```ts
interface EvalSlot {
  readonly containerId: string;       // a container with the /workspace + venv mounts
  readonly docker: DockerHandle;      // the bundle.docker.exec handle (same contract, :2667)
  readonly workspace: string;         // the shared /workspace mount path
  // …a resource budget (cgroup cpu/mem caps) — NO coordinator/policy/MITM/control-server.
}
```

The eval-pool returns `EvalSlot`s, not `DockerInfrastructure`. (If, for implementation convenience, the pool is built atop the existing factory, **explicitly document that the agent-mediation half goes unconstructed** — a `DockerInfrastructure` with null coordinator/proxies — rather than pretending the eval-slot IS a bundle. The clean design returns the narrow `EvalSlot`.)

**Manifest marker.** The `evaluate` `fanOutMember` carries a `schedule` marker so the child's `evaluate` step routes to the eval-pool instead of the shared-container exec:

```yaml
  evaluate:
    type: deterministic
    fanOutMember: true
    resultFile: .evolve_runs/main/current/lane_${laneId}/result.json   # LANE-SCOPED (DA-M3) — not the bare current/result.json
    schedule: { pool: eval, resources: { cpu: 2, mem: 8Gi, gpu: 0 } }   # NEW: route to the eval-slot pool
```

When `schedule.pool` is present, `executeDeterministicState` for that state acquires an `EvalSlot` from the eval-pool (with the declared `resources`) and runs the eval argv through `slot.docker.exec(slot.containerId, …)` rather than `bundle.docker.exec` on the shared agent container; **localhost-first**, the slot is just a cgroup-limited exec on the same host (or a bounded queue sized to the device budget, §0.3 option 3), free over the shared `/workspace` mount.

**Lane-scope the eval `resultFile` (DA-M3).** Today `evaluate.resultFile` is the unscoped `.evolve_runs/main/current/result.json` (`workflow.yaml:428`). With N eval-slots running concurrently, an unscoped `result.json` would **clobber** — all N evals writing one file. The eval writes the **lane-scoped** `current/lane_<k>/result.json`, reconciled with §7.1's lane sub-paths and SF2's host-side containment checks (`applyResultFile`, `orchestrator.ts:2537-2606`, containment at `:2552-2564`) — the pump threads the same lane path into both the in-container write argv and the host-side `input.resultFile`. The **cross-pool coupling is only the candidate file** (researcher writes `current/lane_<k>/result.json` candidate → eval reads it, writes its score back) — free same-host via the shared mount.

**The shared-mount DB-lock safety holds only because of SAME-HOST placement (DA-M3).** §8.3's claim that a second container's `record` is lock-safe rests on the `InterProcessFileLock` (`database.py:209`) being a **file lock on a shared filesystem**. That is true **only because both containers mount the same host `/workspace`** — the OS file lock is meaningful across processes on one host's FS. A cross-host eval-slot writing to a *different* host's FS would NOT see that lock; cross-process DB safety is a **same-host property**, not a free one. So the eval-slot's `record`/`sample` lock-safety is tied to same-host placement specifically (and is exactly why cross-host needs a networked DB store, §8.5).

**Quantify the two-pool cost honestly (DA-S2 — state both directions).** Versus the **whole-lane-bundle** alternative (give each lane its own full `DockerInfrastructure`), two-pool **saves N−1 agent-mediation stacks** — N−1 coordinators + MITM + MCP-relay sets — because the agents stay co-located in the one shared agent container and only `evaluate` fans out. Versus **today's all-shared-container baseline** (one container, N concurrent execs into it), two-pool **ADDS N eval containers**: the container count goes **1 → N+1** (one shared agent container + N eval-slots). The net win is real **because the eval-slots are thin** (DA-M1: no coordinator/policy/MITM) while full bundles are fat — N thin eval-slots cost far less than N−1 fat bundles saved. But the container count *does* grow vs the all-shared baseline; that is the price of resource isolation (§0.3) the all-shared model cannot give.

**Two scheduling levels (note for the scheduler design, not this slice).** There are two distinct granularities: (1) **inter-candidate** — N lanes' evals placed across the eval pool (this slice's concern: the pool runs at most `devices` evals concurrently, rest queue); (2) **intra-evaluation** — one eval fanned across nodes (the evaluator's *own* internal parallelism, e.g. 1000-seed sweeps, `/tmp/alphaevolve.txt:2180`). Level 2 is the evaluator's business; the scheduler just hands it a resource budget and lets it subdivide. This slice schedules at level 1 only.

> **Schema additions (SF1) — `schedule` must be added to the Zod schemas or it is stripped.** `schedule?: { pool: 'eval' | 'agent'; resources?: ResourceBudget }` on `DeterministicStateDefinition` (`types.ts:267-296`) + `deterministicStateSchema` (`validate.ts:78-87`), exactly like `fanOut`/`segment`/`fanOutMember` (§4.3). Unadded, the default `.strip()` drops it silently and every eval routes to the shared container as if `schedule` were absent. The validator should also reject `schedule.pool: eval` on a non-`fanOutMember` state (a non-lane eval has no pool to route to in this slice).

## 9. Cognition store — serialize promotion at the barrier

The cognition store is **NOT** cross-process safe: in-process `RLock` only (`cognition.py:30`), non-atomic `json.dump` save (`:103-112`, `:111`), no file lock, no reload-before-save → last-writer-wins clobber. The search-quality slice's `attach_analysis` promotes the round's lesson via `_promote_lesson` (`evolve_result.py:407`; the `:436` line cited in the first draft is the idempotency-skip check *inside* it — CONSIDER-3 cite drift, corrected). Under N concurrent lanes, N concurrent promotions would clobber the store.

**Resolution for `workers > 1`: move cognition promotion OFF the per-lane record path and onto the serialized barrier join (single writer).** `joinBatch` (§6.2), which runs **after** the barrier (so it is single-threaded by construction) over the children's **done outputs**, promotes each recorded lane's lesson sequentially:

```ts
private async joinBatch(workflowId, outputs: LaneOutput[]): Promise<void> {
  // Single-writer, post-barrier: no concurrent cognition writes possible here.
  // `outputs` are the children's final-state outputs ({status, nodeId|reason, laneId}), §6.2.
  for (const out of outputs) {
    if (out.status !== 'recorded') continue;
    await this.promoteLaneCognition(workflowId, out);    // evolve-cognition add, one at a time
  }
  await this.recomputeStopSignals(workflowId);           // ONE recompute over the N-grown nodes.json
}
```

Concretely, the bridge's per-lane `attach_analysis` is split: the **durable node record** (DB-safe, §8) stays in the per-lane record step; the **cognition promotion** (`_promote_lesson`, search-quality §4.5) moves to a new bridge subcommand `promote_cognition --lane <k>` that the pump calls *only at the join*, serially. The cognition-promoted ledger (`cognition_promoted.json`, search-quality §4.4) is read-modify-written by the single barrier writer, so the plain RMW is correct (no lock needed) — exactly the condition the search-quality slice noted (§4.5: "correct today because the FSM serializes (one writer)"). The barrier preserves that one-writer invariant.

**Stop signals are recomputed ONCE at the join** (`recomputeStopSignals`), over the `nodes.json` that grew by N, not per-lane. The orchestrator reads one fresh `stop_signals.json` per batch (correctness §5.1's post-record vantage, now post-*batch*).

**`workers: 1` MUST keep the legacy inline-promotion path (SHOULD-FIX-5).** The byte-identical no-op gate (item 1) is endangered if §9 *unconditionally* moves promotion to the join even at N=1. Today promotion is **inline** in `attach_analysis` (`evolve_result.py:804` via `_promote_lesson` `:407`): record + promote happen in one call. Moving promotion to a post-barrier `promote_cognition` reorders it to record → (return to FSM) → join-promote. Because cognition feeds the next round's researcher context (the sample step seeds cognition), a promotion-timing change is **observable downstream** and would break byte-identity. **Therefore: at `workers: 1` the pump takes the legacy inline-promotion path (promotion stays in `attach_analysis`); only `workers > 1` routes promotion to the barrier `joinBatch`.** Gate item 1 must run with a **non-empty cognition store** to actually exercise this — an empty store would mask a promotion-timing regression.

**Follow-up noted (not done here):** if a future async slice moves promotion off the barrier (per-record promotion), wrap the promotion + ledger RMW in an `InterProcessFileLock` (importable from the engine's `file_lock` module, `database.py:17`, **not** an engine edit). The bridge already has the import path. This is the async-extensibility hook (§13), deliberately deferred.

## 10. Escalation aggregation — preserve the single-gate model

A lane hitting `evaluator_blocked` (`evaluate`'s verdict, `workflow.yaml:444-445`) — or exceeding per-lane `maxVisits` (§7.5, `lane_max_visits`) — must **not** open a gate mid-batch. There is one `activeGateId` (`orchestrator.ts:2690`), and lanes have no FSM state to match a gate against (`handleGateEntry` is driven by `snapshot.matches(gateName)`, `:1934-1942`), so a per-lane gate is not merely discouraged, it is **unreachable** (CONSIDER C2 — this is why §4.3's "no human-gate inside the fan-out segment" rule is an **error**, not a warning). Resolution: **aggregate at the barrier**.

**Scope correction (do not over-claim):** drain-on-escalation is the **gate-aggregation** mechanism (collapse N potential blocks into one `activeGateId`). It is **NOT** the fix for the held-mutex escalation stall. That stall — an *agent-path policy escalation* awaiting inside the held `callMutex` — is cured separately by **releasing the call mutex across the escalation wait** (§5.2 item 1). The first draft conflated these and wrongly claimed drain "sidesteps the held-mutex stall entirely"; it does not, because the `evaluator_blocked` verdict that trips the drain runs **off-mutex** (`docker.exec`) and never held the mutex. The two mechanisms are now separate: §5.2 fixes the stall; §10 aggregates the gate.

### 10.1 Drain-on-escalation, expressed as a child final state + `childActor.stop()` (CHANGE 1 — read at the barrier, not a manual flag)

The drain *semantics* are unchanged from the first draft; the *mechanism* is now a child final state plus a parent-driven `stop()`, not a hand-rolled `drain` object — and the abort granularity is stated honestly (FSM-M3):

- A lane that hits `evaluator_blocked` (`evaluate`'s verdict, `workflow.yaml:445`) **or** exceeds its own `maxVisits` guard (the child's `isStateVisitLimitReached`, §7.5) transitions the child to a **`blocked` `final` state** whose `output` is `{ status: 'blocked', reason, laneId }` (§6.1). A child whose `agentService`/`deterministicService` *throws* (distinct from the `evaluator_blocked` verdict) routes to the **`errored` `final` state**, `{ status: 'errored', error, laneId }` (§6.1/FSM-S5). The child stops there — XState's normal terminal behavior, no manual `status` mutation.
- The parent learns a lane blocked/errored via that child's final output and **stops the still-running peers** by calling **`childActor.stop()`** on each. **Honest abort granularity (FSM-M3):** `stop()` halts the actor so XState **never enters the child's next deterministic state** — granularity is the **actor-transition boundary**, and it requires **zero executor change**. It does **NOT** thread an `AbortSignal` into the exec layer, because there is none: `deterministicService` / `executeDeterministicState` / `bundle.docker.exec` (`docker-manager.ts:349-354`) take only a **timeout**, and the orchestrator's abort model is a polled `instance.aborted` boolean (`:835`, `:883`, `:916`), not a signal. So `stop()` **cannot** interrupt (a) a queued-but-not-yet-started exec inside a single multi-command `reduceDeterministicCommands` call (the serial loop body at `:2622`), nor (b) a mid-exec `docker.exec` — both run to completion, and the in-flight record commits idempotently (§11.2). Adding an `abortSignal` field through `DeterministicInvokeInput` → `executeDeterministicState` → `docker.exec` for finer-grained mid-exec abort is **explicit future work, not an existing feature.** (The first draft's "`sendStop`/abort threaded into `deterministicService`" language is retracted — it described a capability the exec layer does not have.)
- The barrier (the `Promise.allSettled` join, §6.2) resolves once every child has reached a `final` state (`recorded`, `blocked`, `errored`, or stopped — a stopped child's last snapshot is read as `blocked`/`errored` per why it was stopped).
- `batchVerdict` reads the children's outputs (`Promise.allSettled` results, §6.2/FSM-S5) and returns `{ verdict: 'escalate', items: [<blocked lanes>] }` when any lane is `blocked`, or `{ verdict: 'fail', items: [<errored lanes>] }` when an `errored` lane has no human-actionable recovery (an `errored` lane MAY instead be folded into `escalate` if the operator should decide — the mapping is explicit, not a `Promise.all` reject). The parent FSM transitions `workers → human_escalation` (the orchestrator's existing `escalate` edge, `workflow.yaml:359-360`) on `escalate`, or `workers → failed` (the `result_file_error`-style edge, §4.2) on `fail`. **Exactly one** `human_escalation` gate opens (`handleGateEntry`, `:2685`; `activeGateId = gateRequest.gateId`, `:2690`), carrying the aggregated block reason(s) read from all blocked children's outputs.

This preserves the single-gate model: at most one gate, opened only at the barrier (the parent transition), never mid-lane — and lanes still cannot open a gate themselves, because a child actor has no parent FSM state for `snapshot.matches(gateName)` to fire against (`:1934-1942`), exactly the unreachability invariant §4.3 enforces as an error.

### 10.2 The richer option (noted, not first cut)

A **queued/aggregated multi-item gate**: instead of draining the whole batch on the first block, let lanes that *can* finish do so, collect all blocked lanes, and present a multi-item gate ("3 of N lanes blocked: lane 0 …, lane 2 …"). This needs the gate DTO to carry a list of block items and the resume path to re-run only the blocked lanes. It is more efficient (non-blocked lanes' work is not discarded) but touches the gate surface (`activeGateId` is still single, but the gate *payload* grows). **Recommendation: drain-on-escalation first; the multi-item gate is a later enhancement** (§14 OQ3). Both preserve `activeGateId` as a single string — the difference is whether non-blocked lanes' in-flight work is kept.

### 10.3 Resume after escalation

After the human resolves the gate (`APPROVE` → `orchestrator`, or `FORCE_REVISION` → `preflight`), resume re-runs the drained batch (§11). Because records are idempotent (correctness §6.2) and DB-keyed (§8), the lanes that *did* record before the drain are skipped; the drained/blocked lanes re-run from `sample`. This is the same machinery as crash-resume.

## 11. Checkpoint / resume

The barrier is a clean single-active-state checkpoint boundary (§2.2). Between batches the FSM checkpoint is flat (`machineState: "orchestrator"` or `"workers"`). The hard case is a crash *mid-batch*. Resolution leans entirely on the shipped container-snapshot machinery + **DB-as-truth** (`workflow-container-snapshot-resume.md`): durable state lives outside the FSM (the engine DB on the host `/workspace` mount; container env in the snapshot), so resume = restore + re-run batch, with **no per-lane logical FSM-checkpoint surgery**.

### 11.1 The checkpoint shape for an in-flight batch (shape A makes resume work)

- **FSM checkpoint:** `machineState: "workers"` — a **real** state (shape A, §4.2). This is the load-bearing reason for shape A: resume does `definition.states[String(checkpoint.machineState)]` and reads `.type` (`orchestrator.ts:1558-1561`), which **only works if `workers` is a declared state.** XState's `resolveState` restores *to* `workers` but does not *enter* it (`:1554-1556`), so `replayInvokeForRestoredState` (`:1571`) re-runs the `workers` invoke — i.e. **re-runs `runFanOutSegment`**, which re-spawns the children. Under the rejected shape B, `machineState` would be `sample` and resume would re-run one bare single-`current/` `sample` exec, not the fan-out (FSM review MF1).
- **Lane reconstruction is DB-derived; no `batch.json` exists (CHANGE 1, §6.3):** on resume, `runFanOutSegment` reconstructs the batch purely from the engine DB — a lane whose `step_<batch_index>_lane_<k>` is already a recorded node (`_find_recorded_node_for_step`, `evolve_result.py:186-191`) is **not re-spawned** (its `recorded` output is synthesized); every other lane is spawned fresh from `sample`. The engine DB is the sole authority. **There is no advisory `batch.json` to trust or distrust — it is deleted (§6.3).** `batch_index` is recovered from the DB as `1 + max(batch_index in any recorded step_name)` (§6.3).
- **Persistence-caveat reminder (§6.3):** the first cut does NOT persist live mid-batch child actors. The current checkpoint is `{ machineState, context }` via `resolveState` (`:1457`), which does not rehydrate spawned-child `ActorRef`s, so resume relies on **DB-as-truth + re-spawn**, not on XState's `getPersistedSnapshot()`. Migrating the checkpoint to `getPersistedSnapshot()` (to restore in-flight children directly) is the clean long-term option flagged for reviewers in §6.3, not assumed here.

### 11.2 Why re-running the batch is safe (idempotency) — and what it can still change

The correctness slice made `analysis_record` idempotent (keyed on `meta_info.step_name`, `:732-733`, `:186-191`): re-recording a lane's step is a no-op that returns the existing node id. With lane-tagged step names (§7.2), each lane's record is independently idempotent. So re-running a partially-completed batch:

- re-runs `sample`/`researcher`/`evaluate` for every non-recorded lane (evals are idempotent — they recompute the same score for the same candidate),
- re-records nodes only for lanes that did not record before the crash (recorded lanes short-circuit),
- leaves **no duplicate nodes** (gate item 6).

The agent re-generation (researcher/analyzer re-run produce a *fresh* candidate, possibly different from the pre-crash one) is **acceptable for node-set integrity**: a crashed lane had no recorded node, so its pre-crash candidate was never durable. (Fully reproducible re-generation would require persisting the agent's output per lane before record — out of scope.)

**But resume can change a batch's VERDICT, not just its node set (SHOULD-FIX-11 / FSM review SF4).** A re-created lane's child generates a *fresh* candidate; if that fresh candidate hits `evaluator_blocked` where the pre-crash one would not have (or vice-versa), the resumed batch **drains and escalates even though the pre-crash batch would not have** (or escalates differently). The barrier (the `Promise.allSettled` join) only resolves when all children reach a `final` state (`recorded`/`blocked`/`errored`), so one re-created lane's different child output flips the whole batch verdict. **Consequence stated plainly:** gate item 6 ("no duplicate nodes") still passes while the batch *verdict* silently differs run-to-run across a resume. This is a correctness-of-batch-outcome drift, not a node-integrity bug. If batch-verdict stability matters for a given run, the pre-crash candidate must be persisted per lane before record — explicitly out of scope here, but the drift consequence is now on the record.

### 11.3 Container restore

Resume mints a fresh container (lazy, `orchestrator.ts:1527-1531`) and, when the checkpoint carries snapshot digests, restores them by immutable digest (`workflow-container-snapshot-resume.md`). The `/workspace` mount (DB) is reattached, so durable state survives. **The bundle is re-minted once by `runFanOutSegment`'s pre-fan-out `ensureBundleForScope` (§6.2/§8.1), before any child is re-spawned** — the same mint-before-fan-out discipline as a fresh start, so the resume path also avoids the N-container mint race. No per-lane container state exists (lanes share one container — or, with the §8.6 eval pool, the eval-slot bundles are re-requested from the scheduler factory on the re-spawn), so there is nothing lane-specific to restore beyond the shared workspace.

## 12. Reproducibility

The search-quality slice added seeded samplers (`sampling_seed.txt`) for reproducibility. Concurrent lanes make **DB-write order timing-dependent**: if two lanes record at nondeterministic relative times, their node ids (engine-assigned in lock-acquire order, `database.py:73-74`) and thus the `nodes.json` ordering can differ run-to-run.

**For sync, the mitigation is: serialize the DB-mutating critical section (`sample` parent-selection + `record`) through a deterministic order at the join, while `design`/`eval` run in parallel.** Two grades:

- **(a) Parallel everything, accept id-order nondeterminism.** Let all N lanes record concurrently (DB-safe, §8); accept that node ids 21/22/23 may be assigned to lanes 0/1/2 in any order across runs. The *set* of nodes is deterministic (same candidates, same scores, given seeds); only the id↔lane assignment varies. Cheapest; reproducible at the *population* level, not the *id-ordering* level.
- **(b) Serialize sample+record in lane order at the barrier-adjacent steps.** Run `researcher`/`evaluate`/`analyzer` (the expensive, parallelizable work) concurrently, but funnel the `sample` parent-selection and the `record` node-append through a **lane-ordered critical section** (lane 0 records before lane 1, etc.) so node ids are assigned in lane order deterministically. This keeps `design`/`eval` parallel (the speedup) while making DB-mutation order deterministic (the reproducibility).

**Recommendation: state the trade-off, default to (a), offer (b) under a `reproducible: true` flag.** (a) gives the full parallel speedup; (b) costs a serialization point at the join but gives byte-identical id-ordering across runs — valuable for CI/regression gates. The `workers: 1` backward-compat gate (item 1) is unaffected either way (one lane = no ordering question).

**Honest reproducibility statement — narrowed (SHOULD-FIX-3).** The first draft over-claimed "reproducible as a population (same nodes, same best score, given seeds)." That is wrong in two ways: (1) **candidate *content* is not reproducible in either the linear OR the parallel chain** — the agents (researcher/analyzer) are LLM calls and are not seeded; `PYTHONHASHSEED=0` (`evolve_result.py:17`) seeds `hash()` only, not the LLM and not Python `random`. The search-quality slice's `sampling_seed.txt` makes only the **sampler's parent selection** reproducible, never the candidates. (2) **Resume re-generation makes a resumed run diverge from a clean run regardless of N** (§11.2). So the correct, narrow claim is:

- The **sampler's parent selection** is reproducible given seeds (and, with the barrier-side `sample(n=N)` partition + per-lane seed `base_seed + laneId` of §7.4, lane-diverse-but-seed-determined).
- **Candidate content is not reproducible** in either the linear or parallel chain (LLM nondeterminism). The parallel slice does not *break* a reproducibility the linear chain had — it inherits a claim that was only ever about the *sampler*.
- `reproducible: true` (option b) buys byte-identical **id-ordering** (which lane's candidate gets which node id), **not** byte-identical *candidates*. Say so on the gate.

## 13. Async-extensibility — what stays, what changes

**Sync is genuinely a stepping stone for these domains, not the end state.** AlphaEvolve is async *by construction*: its pipeline is *"an asynchronous computational pipeline … many computations are run concurrently, with each computation blocking whenever its next step relies on the result of another, yet unfinished computation … optimized for throughput (rather than the speed of any one particular computation)"* (`/tmp/alphaevolve.txt:455-462`), and evaluation is distributed via *"asynchronous calls to an evaluation cluster"* (`:425-426`). The hard barrier this slice adopts (the `Promise.allSettled` join, §6.2) pays a straggler tax every batch (§0.1 fact 3), and that tax is **large** for the target domains precisely because real evaluation is high-variance — cascade evaluation prunes weak candidates fast while survivors run orders of magnitude longer (ASI-Evolve v2's 300s eval / 1800s engineer timeouts, `config.yaml:20`,`:61`, are the order-of-magnitude spread). **The straggler tax is acceptable for this slice only because the chosen validation domain (§14 OQ6, the LLM-serving scheduler) has *bounded* eval variance** — a fixed-trace simulation whose per-candidate runtime spread is narrow. For the truly high-variance flagship domains, the async follow-up (which overlaps the straggler with other lanes' next candidates) is where the throughput win is fully realized; sync establishes the child-machine lane machinery, the resource-model hook (§0.3 / §8.5), the two-pool split (§8.6), and the DB/cognition safety that async reuses. The N-lane machinery is shaped so async is a **later additive step, not a rewrite**.

**Async is a DIAL — pool depth — and the eval-pool is the concrete seam where it re-enters (DA-C1).** "Sync vs async" is not a clean either/or boundary this slice cleanly sits inside; it is a **continuum the eval-pool already parameterizes.** The moment the §8.6 eval-pool is given a device cap **< N** (the realistic case — fewer GPUs/cores than lanes, §0.3), it is **already a bounded async work-queue**: at most `devices` evals run, the rest queue, and lanes complete out of order as slots free. So async re-entry is not a future rewrite of the barrier — it is **turning the same pool's depth knob** and replacing the all-N `Promise.allSettled` join with a tick-on-each-completion controller that reads the queue as evals finish. The eval-pool is therefore the **concrete seam where async first re-enters the design**; "this slice is sync" really means "this slice runs the pool at depth N with a hard join," not "this slice is architecturally async-incapable." The async follow-up (controller-tick-on-record + bounded pool) reuses:

| Piece | Sync (this slice) | Async (later, additive) |
| --- | --- | --- |
| Round sub-machine + child actors (§6.1) | N free-standing children created, all run to barrier | Same child machine; pool creates child K+1 when a slot frees |
| `runFanOutSegment` create+barrier (§6.2) | `createActor` ×N, `Promise.allSettled` joins | Create under the eval-pool at depth < N; the join becomes tick-on-completion-then-backfill |
| `current/lane_<k>/` working areas (§7) | N fixed lanes | Same; pool reuses a freed lane's dir |
| Lane-tagged step names (§7.2) | `step_<batch>_lane_<k>` | Same; `batch` becomes a logical tick id |
| Per-lane FSM bookkeeping (child context, §6.1/§7.5) | Native per-child within a batch (visit counts, maxVisits, tokens); resets on resume (FSM-M1) | Native per-child — same within-batch property; crash-durability still needs `getPersistedSnapshot()`+`spawn` (§6.3) |
| Scheduler-factory + eval-pool lifecycle (§8.5) | Localhost emulation factory; pool is a new acquire/release/queue lifecycle (DA-S3) | Distributed factory (eval nodes) — same factory contract; same pool lifecycle, depth < N |
| Two-pool eval scheduling — thin `EvalSlot`, not a bundle (§8.6) | `schedule: {pool: eval}` → localhost cgroup-limited `EvalSlot`s | Same marker → distributed eval-node pool of `EvalSlot`s |
| DB cross-process safety (§8) | Already safe (SAME-HOST file lock, DA-M3) | Needs a networked DB store once eval-slots go cross-host (§8.5) |
| Cognition promotion (§9) | Serialized at barrier (single writer); inline at `workers:1` | Per-record promotion + `InterProcessFileLock` (the deferred hook, §9) |
| Escalation as child `blocked`/`errored` final (§10) | Read at barrier; `childActor.stop()` halts peers at actor-transition boundary (FSM-M3) | Child `blocked`/`errored` output stops the pool launching, aggregates |
| DB-derived lane reconstruction (§6.3/§11.1) | From recorded `step_<batch>_lane_<k>` nodes | Same; `batch` becomes a logical tick id |
| Mutex-release escalation fix (§5.2) | Releases `callMutex` across human wait | Reused unchanged — now N escalations can be live |
| Resource model at the `evaluate` exec point (§0.3) | Shared container suffices for CPU+memory-bound eval (option 0); CPU/mem caps available, GPU isolation deferred (DA-S1) | The bounded eval queue / per-device isolation IS the on-ramp to the paper's distributed evaluator cluster (`:455-462`) |
| Barrier (the `Promise.allSettled` join) | Hard join per batch (= eval-pool at depth N) | Same pool at depth < N + controller-tick-on-completion (the one structural change = a dial, DA-C1) |

The **only** structural change async requires is replacing the hard barrier (the `Promise.allSettled` over all N children) with a controller that ticks on each child's record and runs the **same eval-pool at depth < N** (backfilling a fresh child when a slot frees) — and as DA-C1 makes explicit, that is **turning the pool's depth knob, not rebuilding the join.** Everything else (the round child machine, per-lane working areas, step naming, native per-child bookkeeping, DB safety, escalation-as-`blocked`/`errored`-final, DB-derived lane reconstruction, the §8.5 factory + pool lifecycle, and the §8.6 thin-`EvalSlot` two-pool split) is reused. The cognition `InterProcessFileLock` and the per-record stop-signal recompute are the two bridge-side additions, both engine-untouched. **This is the explicit promise: sync is not a dead end** — and the §8.5 factory/pool seam plus §0.3's resource model (CPU/mem isolation now, per-device GPU isolation as the hardware-dependent follow-up) is the same hook that async needs to reach the paper's distributed evaluator cluster for the GPU/TPU domains.
