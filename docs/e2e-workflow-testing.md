# End-to-End Workflow Web UI Testing

How to run a real or mock-based E2E test of the workflow web UI.

---

## 1. Mock-Based Manual Test (No Docker / No API Key Required)

The fastest path. Uses the mock WebSocket server to simulate the full workflow lifecycle. Good for frontend development, UI validation, and gate interaction testing.

### Prerequisites

- Node.js 20+
- `npm install` at repo root (installs workspace deps)

### Steps

```bash
# Terminal 1: Start mock WebSocket server (port 7400)
cd packages/web-ui && npm run mock-server

# Terminal 2: Start Vite dev server (port 5173, proxies /ws to 7400)
cd packages/web-ui && npm run dev
```

Open `http://localhost:5173?token=mock-dev-token` in your browser.

### What you can do

1. **Navigate to the Workflows page** (sidebar)
2. **See two pre-loaded workflows**: "design-and-code" (running) and "code-review" (waiting at gate)
3. **Click a workflow** to see the state graph visualization, transition history, context info
4. **Start a new workflow**: Click "Start Workflow", pick a definition, enter a task description. The mock server will simulate state transitions with realistic timing (plan -> plan_review gate in ~3s)
5. **Resolve a gate**: When a workflow reaches a human gate (orange highlight), the gate panel appears. You can Approve, Force Revision (with feedback text), or Abort
6. **Browse workspace files**: File tree and content viewer show canned data
7. **View artifacts**: Artifact panel shows canned plan/spec content

### Limitations

- No real agent execution -- state transitions are scripted with delays
- File tree/content is canned, not from a real workspace
- Artifacts are hardcoded, not generated by an LLM
- Workflow completes via scripted timer, not real work

## 2. Real E2E Test (Full Stack)

Runs the actual daemon, creates real Docker agent sessions, runs real LLM agents, and pushes real lifecycle events through the WebSocket to the browser.

### Prerequisites

1. **Docker** installed and running (the bundled `design-and-code` workflow uses `mode: "docker"` with `dockerAgent: "claude-code"`)
2. **`ANTHROPIC_API_KEY`** set in `.env` at repo root (or exported in shell)
3. **Compiled policy**: `npm run annotate-tools` and `npm run compile-policy` must have been run at least once (generates `src/config/generated/compiled-policy.json` and `tool-annotations.json`)
4. **Built project**: `npm run build` (compiles TypeScript + builds web UI static assets to `dist/web-ui-static/`)

### Verify prerequisites

```bash
# Check Docker
docker info >/dev/null 2>&1 && echo "Docker: OK" || echo "Docker: NOT AVAILABLE"

# Check API key
[ -n "$ANTHROPIC_API_KEY" ] && echo "API key: set" || echo "API key: NOT SET (check .env)"

# Check compiled policy
[ -f src/config/generated/compiled-policy.json ] && echo "Policy: compiled" || echo "Policy: NOT COMPILED"

# Check web UI build
[ -f dist/web-ui-static/index.html ] && echo "Web UI: built" || echo "Web UI: NOT BUILT"
```

### Option A: Production mode (served from dist/)

```bash
# Build everything
npm run build

# Start daemon with web UI
node dist/cli.js daemon --web-ui --no-signal
```

The daemon prints the URL with auth token:
```
IronCurtain daemon started.
  Control socket: listening
  Web UI: http://127.0.0.1:7400?token=<RANDOM_TOKEN>
```

Open that URL in your browser.

### Option B: Dev mode (Vite HMR + real daemon)

```bash
# Terminal 1: Start daemon with web UI in dev mode
npx tsx src/cli.ts daemon --web-ui --web-ui-dev --no-signal

# Terminal 2: Start Vite dev server
cd packages/web-ui && npm run dev
```

The daemon prints:
```
  Web UI: http://127.0.0.1:7400?token=<TOKEN>
  Web UI (dev): http://localhost:5173?token=<TOKEN>
```

Open the dev URL (`localhost:5173`) for HMR support.

### Running a workflow

1. Open the web UI in your browser
2. Navigate to the **Workflows** page
3. Click **Start Workflow**
4. Select `design-and-code` from the definition dropdown (auto-discovered from bundled workflows)
5. Enter a task description (e.g., "Build a CLI calculator in TypeScript")
6. Optionally specify a workspace path (leave blank for auto-created temp dir)
7. Click Start

### What happens

1. **`workflows.start` RPC** is received by the daemon's `WorkflowManager`
2. `WorkflowManager.getOrchestrator()` lazily creates a `WorkflowOrchestrator` using `createWorkflowSessionFactory()` -- this loads `IronCurtainConfig` and wires up `createSession()`
3. The orchestrator loads the workflow definition, creates an xstate machine, and starts executing
4. For each `agent` state:
   - A Docker session is created (Claude Code in a container with `--network=none`)
   - The agent receives the state's prompt + input artifacts
   - Real LLM calls happen, real tool calls are mediated by the policy engine
   - When the agent completes, output artifacts are collected
5. **Lifecycle events** (`workflow.state_entered`, `workflow.agent_started`, `workflow.agent_completed`) are emitted to the `WebEventBus` and pushed to connected WebSocket clients
6. When a `human_gate` state is reached:
   - `workflow.gate_raised` event is emitted with the gate details (presented artifacts, accepted events, summary)
   - The web UI shows the gate panel with Approve/Force Revision/Replan/Abort buttons
   - The user reviews artifacts (real files from the workspace) and clicks a button
   - `workflows.resolveGate` RPC sends the decision back
   - The orchestrator continues to the next state
7. On completion, `workflow.completed` event fires

### Expected timeline for design-and-code

| State | What happens | Approx. time |
|-------|-------------|--------------|
| `plan` | Agent writes `.workflow/plan/plan.md` | 30-90s |
| `plan_review` | **Gate**: Review the plan, click Approve | Manual |
| `design` | Agent writes `.workflow/spec/spec.md` | 30-90s |
| `design_review` | **Gate**: Review the spec, click Approve | Manual |
| `implement` | Agent writes code in workspace root | 60-180s |
| `review` | Critic agent reviews implementation | 30-90s |
| `implement` (loop) | If rejected, coder fixes issues | 60-180s |
| `final_review` | **Gate**: Review final code, click Approve | Manual |
| `completed` | Done | -- |

Total: 5-15 minutes depending on task complexity and LLM response times.

## 3. Architecture: How It All Connects

```
Browser (Svelte SPA)
   |
   | WebSocket (JSON-RPC + events)
   | Bearer token auth via ?token= query param
   |
WebUiServer (src/web-ui/web-ui-server.ts)
   |
   |-- dispatch() routes "workflows.*" methods
   |       |
   |       v
   |   WorkflowManager (src/web-ui/workflow-manager.ts)
   |       |
   |       |-- getOrchestrator() [lazy init]
   |       |       |
   |       |       |-- createWorkflowSessionFactory() [from cli-support.ts]
   |       |       |       |-- loadConfig()
   |       |       |       |-- returns (opts) => createSession(opts)
   |       |       |
   |       |       |-- FileCheckpointStore (temp dir)
   |       |       |
   |       |       v
   |       |   WorkflowOrchestrator (src/workflow/orchestrator.ts)
   |       |       |-- xstate machine per workflow
   |       |       |-- creates Docker sessions per agent state
   |       |       |-- emits lifecycle events
   |       |
   |       |-- forwardLifecycleEvent() -> WebEventBus -> broadcast to WS clients
   |       |-- raiseGate() -> WebEventBus "workflow.gate_raised"
   |
WebEventBus -> broadcast() -> all connected WS clients
```

### Key wiring in the daemon (`ironcurtain-daemon.ts`, lines 666-693):

```typescript
// startWebUiServer():
const server = new WebUiServer({ ... });
server.setWorkflowManager(new WorkflowManager({ eventBus: server.getEventBus() }));
```

The `WorkflowManager` is created with the server's `WebEventBus`. When the orchestrator emits events, they flow through the manager's `forwardLifecycleEvent()` into the bus, which broadcasts to all connected WebSocket clients.

The `WorkflowManager.createOrchestrator()` method:
- Creates a temp dir for checkpoints
- Calls `createWorkflowSessionFactory()` which loads config and returns a session factory
- Wires up gate raise/dismiss callbacks to emit `WebEventBus` events
- Creates no-op tab handles (mux tabs are meaningless in web UI mode)

## 4. Known Gaps and Issues

### No gaps blocking the real E2E -- it should work as-is

The wiring is complete:

- `ironcurtain-daemon.ts` creates `WorkflowManager` with the event bus (line 684)
- `WorkflowManager.createOrchestrator()` uses `createWorkflowSessionFactory()` which calls `loadConfig()` and `createSession()` (the same path as CLI workflows)
- The orchestrator creates real Docker sessions based on the definition's `mode: "docker"` setting
- Lifecycle events are forwarded through `WebEventBus` to WebSocket clients
- `workflow-dispatch.ts` handles all `workflows.*` RPC methods
- Gate resolution via `workflows.resolveGate` calls `controller.resolveGate()`

### Things to be aware of

1. **Docker is required for the bundled workflow**. The `design-and-code.json` workflow has `"mode": "docker"`. If Docker is not available, the session factory will fail. To test without Docker, you would need a custom workflow definition with `"mode": "builtin"` -- but builtin mode uses the code sandbox, not a full agent, so the experience would be different.

2. **Cost**: Each agent state invocation runs a real LLM session. The `design-and-code` workflow has 4+ agent states, and the implement/review loop can repeat up to 3 times. Budget: expect $2-10 per full run depending on task complexity.

3. **The `--no-signal` flag is recommended** for testing. Without it, the daemon tries to start the Signal transport, which requires Docker + Signal container setup.

4. **The mock server token is hardcoded** (`mock-dev-token`). The real daemon generates a cryptographically random token on each start. Copy it from the terminal output.

5. **Web UI build**: If you change frontend code in `packages/web-ui/src/`, either use dev mode (Option B above) or rebuild with `npm run build:web-ui` before restarting the daemon.

## 5. Troubleshooting

| Problem | Cause | Fix |
|---------|-------|-----|
| "Web UI not built" in browser | `dist/web-ui-static/` missing | `npm run build` or `npm run build:web-ui` |
| WebSocket 401 | Wrong or missing token | Copy the full URL with `?token=...` from daemon output |
| "Workflow system not available" RPC error | `--web-ui` flag not passed | Start daemon with `ironcurtain daemon --web-ui` |
| Session creation fails | Docker not running or no API key | Check `docker info` and `.env` |
| Workflow stuck at agent state | LLM taking long or rate limited | Check daemon logs at `~/.ironcurtain/logs/daemon-*.log` |
| "No output generated" from agent | Budget exhausted or model error | Check session logs, increase budget in `~/.ironcurtain/config.json` |
| Gate not appearing in UI | WebSocket disconnected | Check browser console for WS errors; refresh page |
| CORS/Origin error on WebSocket | Using Vite dev server without `--web-ui-dev` | Add `--web-ui-dev` flag when using Vite dev proxy |
| Port 7400 in use | Previous daemon still running | `ironcurtain daemon status` to check; kill the old process |

### Viewing logs

```bash
# Daemon log (stderr redirected to file after startup)
tail -f ~/.ironcurtain/logs/daemon-*.log

# Session logs (one per agent invocation)
ls ~/.ironcurtain/sessions/*/session.log
```
