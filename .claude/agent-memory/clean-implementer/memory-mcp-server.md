# memory-mcp-server package (packages/memory-mcp-server)

## Layering & seams
- `MemoryEngine` interface (`src/engine.ts`) is the public surface; tool handlers receive ONLY the engine, never `MemoryConfig`. LLM/config + `db` are captured in the engine closure in `src/engine-impl.ts` (`createMemoryEngineFromConfig`). New write-side LLM features (extraction) belong on the engine, parallel to `store`/`recall`.
- `EngineModules` (test factory `createMemoryEngine`) must mirror every `MemoryEngine` method. Adding a method means updating: interface, `createMemoryEngineFromConfig`, `EngineModules`, `createMemoryEngine` passthrough, AND every test mock engine (`test/server.test.ts`, `test/tools.test.ts`, plus any new tool test).
- `llm/client.ts` exports `llmComplete(config, system, user, {maxTokens})` (returns `null` on no-LLM or error) and `getLLMClient` (null when both `llmApiKey` and `llmBaseUrl` are null). Parse JSON out of text defensively — see `parseBatchJudgments`. NO structured-output/function-calling API anywhere; NO retry loops.
- `estimateTokens` lives in `retrieval/scoring.ts` (~4 chars/token). `MAX_CONTENT_LENGTH=10000` in `tools/validation.ts`. `EXACT_DEDUP_DISTANCE=0.05` in `storage/constants.ts`.

## Parent-context retrieval ("index fine, return coarse", schema v4)
- `segments` table + `memories.segment_id` FK (nullable) added to canonical `createSchema` (NO migration runner). `SCHEMA_VERSION='4'`; `initDatabase` calls `dropStaleSchema(db)` BEFORE `createSchema` — reads on-disk `schema_meta['schema_version']` defensively (absent table ⇒ fresh, no drop). Segments are OFF the index (no vec/FTS row, no trigger).
- **`dropStaleSchema` compares NUMERICALLY (fixed F5, was string `!==`):** `Number(onDisk)` vs `Number(SCHEMA_VERSION)`. absent stamp→no-op; `<`→`rebuildSchema` (drop+recreate, discard old data); `===`→no-op; `>`→**THROW** (`/newer than/`, fail closed — an older binary must NOT destroy a newer DB it can't read); NaN/unparseable stamp→rebuild (treated as stale, not a recognizable newer version). The drop body was extracted to a `rebuildSchema(db)` helper. Test for the `>` case: build a v4 DB, insert a row, `UPDATE schema_meta SET value='5'`, close; assert `initDatabase` throws AND reopen raw via `new Database(path)` to confirm the stamp+row both survived (initDatabase never reaches createSchema on throw).
- `expand: 'none'|'auto'|'parent'` on `RecallOptions`, default `'auto'`. Auto fires when ≥2 kept facts share a non-null `segment_id`. Step 9b lives in `retrieval/expansion.ts` (`expandKeptFacts`) — inserted in `pipeline.ts` AFTER `deduplicateByEmbedding`, BEFORE `packToBudget`. Ranker (steps 1–9) byte-for-byte unchanged; `'none'` is pure pass-through.
- **AUGMENT, not REPLACE (the corrected semantics).** `buildDisplayList` keeps EVERY kept fact (identical to `expand:'none'`) and APPENDS passages AFTER all facts, in segment-best-rank order. Earlier REPLACE design collapsed sibling facts into one passage → destroyed breadth on multi-term sources (contract "key terms" survey query was WORSE than `none`). Because the unchanged greedy skip-not-break `packToBudget` packs in order, passages-last means facts pack first and a passage only takes leftover budget → **auto-expand can NEVER evict a fact `none` would have kept** (load-bearing; tight budget ⇒ auto≈none, the passage is the unit skipped). Parent-dedup now applies to PASSAGES (one per shared segment), never to facts.
- **id-collision gotcha (the subtle bug AUGMENT introduces):** the appended passage unit rides on its segment's best-ranked fact (`groups.get(segId)[0]`) for date/importance — but that fact is ALSO emitted, so the passage MUST get a DISTINCT id (`${bestFact.id}#seg:${segId}`). A shared id makes id-keyed structures treat the passage as a duplicate of its host fact and SILENTLY DROP it: `clusterByEmbedding` (extractive summary) skips by `assigned.has(id)`, and `updateAccessStats` double-counts. The synthetic id has no `memories` row so access-stat UPDATE no-ops on it (correct). `selectedEmbeddings` keyed by id ⇒ passage falls into its own singleton cluster (fine). `memory_ids` from the pipeline leaks the synthetic id but is NOT surfaced in public `RecallResult` (only `memories_used`/`total_matches`/`expanded`/`expanded_segment_ids`).
- Returned unit on expansion is a query-ranked ~350-token PASSAGE (`splitToPassages` in `extraction.ts`, cap `MAX_PASSAGE_TOKENS=350`), ranked by cosine to the REUSED step-1 query embedding (no 2nd query embed). `packToBudget` was made generic `<T extends ScoredMemory>` so it preserves the `DisplayUnit` subtype (non-behavioral). Default `MEMORY_DEFAULT_TOKEN_BUDGET` bumped 500→800.
- The `max_expand_passages` cap moved: it is NO LONGER in `choosePassages` (which now computes a passage for every candidate segment, no cap) — it caps SURVIVING passages in `buildDisplayList`'s append loop (`expandedSegmentIds.length >= maxExpandPassages` break), AFTER overlap-dedup, so a passage dropped as an overlap duplicate does not waste a cap slot.
- `MemoryEngine.expand(segmentId, query?)` powers the `memory_expand` tool (7 tools now; `server.test.ts` asserts the list). Adding it touched: interface, factory, `EngineModules`, `createMemoryEngine`, + every mock engine (server/tools/ingest-tool tests).
- TESTING the passage split-and-rank with the real BGE embedder: synthetic clause paragraphs must each be ≥~200 tokens so the greedy `splitToPassages` packer keeps them as DISTINCT passages (else short clauses merge into one passage under the 350 cap and the query-relevance assertion fails). The BGE QUERY PREFIX matters: a query token that also appears in a generic overview passage (e.g. "Bandalert") can pull the overview above the topical clause — keep discrimination queries topical, drop the entity name. Forgotten-parent test: `db.pragma('foreign_keys=OFF')` around the segment DELETE to leave a genuine dangling pointer (FK is ON by default).
- TWO co-retrieving shared-parent segments (needed for overlap-dedup §5.3.2 and the max_expand_passages cap §5.4, which both require >1 segment auto-expanding in one result): ingest each as a SEPARATE `engine.ingest()` call (→ 1 segment, ≥2 facts each); give both an OFF-query topical domain so a single query co-retrieves facts from both (e.g. two vendor SLAs sharing "service level agreement uptime credits"). VERIFIED to co-retrieve reliably with `rerankerEnabled:false`. Assert the precondition (`segments_created===1` + `created===2` per ingest, `total_matches>=4`) so the test can't pass vacuously.
- `isOverlapping` (`expansion.ts`) is a FULL-PASSAGE (near-)substring check, NOT sentence-level: it only fires when one chosen passage is a substring of another. To trigger it: put an IDENTICAL clause as its own paragraph in both segments AND pad each segment's OTHER (overview) paragraph past the 350-token cap so `splitToPassages` ISOLATES the shared clause as its own passage (`OVERVIEW_PAD.repeat(5)` ≈ 315t overview + 105t clause → 2 passages). Then a query targeting the shared clause makes split-and-rank pick the identical passage in both → dedup drops the 2nd → `expanded_segment_ids.length===1`. Mutating `isOverlapping → false` flips it to 2 (non-vacuity confirmed).
- For the CAP test specifically use DISTINCT chosen passages (query each segment's distinct overview) so overlap dedup is NOT what limits the result — only the cap is. Control with `max_expand_passages:5` (asserts both expand, length 2) then cap at 1 (length 1). Mutating away the `expandedSegmentIds.length >= maxExpandPassages` break in `buildDisplayList` (post-AUGMENT location) flips the capped case to 2 (non-vacuity confirmed).
- **Breadth-not-evicted regression test (the AUGMENT load-bearing guard):** compare `expand:'auto'` vs `expand:'none'` FACT-unit content sets (parse `format:'raw'`, split on the `expanded` flag) at the SAME budget. Wide budget ⇒ auto facts ⊇ none facts AND a passage appended. The DISCRIMINATING tight budget is `passageTokens + allFactTokens - 1` (measure both off a wide auto raw recall): facts-last fits all facts and skips the passage by 1 token; a HIGH-priority passage would fit first and evict ≥1 fact (set becomes a strict subset). VERIFIED the test FAILS under a passages-prepended mutation (`Set{2}` vs `Set{3}`) and passes correctly. A naive tight budget (`factsTokens+40`) does NOT discriminate — the passage never fits either way, mutant passes vacuously.
- A4 merge-repoint (`updateMemorySegmentIfRicher`) branches: TIE keeps existing — ingest two distinct segments each `fact_count=N` (the segment's `factCount` = `group.facts.length`, counted BEFORE dedup, so a 2-fact group whose 1st fact merges still stamps `fact_count=2`), the duplicate's merge keeps the ORIGINAL `segment_id`. NULL-adopt — `engine.store()` (segment_id NULL) then `engine.ingest()` a duplicate carrying a segment: the exact-dedup merge crosses the store→ingest boundary (identical content → distance 0) and repoints NULL→incoming. Read survivors back via `getMemoriesByIds`.
- **F2 — MCP `memory_recall` surfaces expansion metadata via `structuredContent`:** `@modelcontextprotocol/sdk@1.27.1`'s base `CallToolResultSchema` carries an OPTIONAL `structuredContent: z.record(z.string(), z.unknown())` — no `outputSchema` registration required, so the deprecated `server.tool()` callback can just return `{ content:[{type:'text',text}], structuredContent:{...} }`. `handleRecall` (`tools/recall.ts`) now returns `RecallToolResult {text, memories_used, total_matches, expanded, expanded_segment_ids}` (NOT a bare string); `server.ts` puts `text` in the text block and the 4 metadata fields in `structuredContent` (defaults `expanded:false`/`[]` from the optional `RecallResult` fields). Empty case keeps `text:'No relevant memories found.'`. Tool-level test asserts `result.structuredContent` (NOT just engine-level) in `server.test.ts` — that was the review's coverage blind spot.

## Extraction content cap (F7)
- `coerceFact` (`storage/extraction.ts`) now routes BOTH the bare-string branch AND the object branch through a shared `capFactText(text)` = `trim()` then `slice(0, MAX_CONTENT_LENGTH)`. Before F7 only the object branch capped; a model response shaped `["<very long>"]` bypassed it. Test mirrors the object-fact cap test with a bare-string `JSON.stringify([longString])` asserting `facts[0].fact.length === MAX_CONTENT_LENGTH` (`MAX_CONTENT_LENGTH=10000` from `tools/validation.ts`).

## IngestResult shape
- `IngestResult` (`src/types.ts`) fields: `created`, `merged`, `memory_ids`, `facts`, optional `chunks`/`failed_chunks`/`degraded`/`partial`/`skipped`. The `ingested` alias (== `created`) was REMOVED on feat/memory-ingest — it was dead (only test assertions read it; corpus driver's `IngestResultLike` reads `created`/`merged`, not `ingested`). Do not reintroduce it.
- `ingestBlob`'s `facts.length === 0` "nothing written" returns share an `emptyResult(extra?)` local helper (`{created:0,merged:0,memory_ids:[],facts:[],...extra}`); used for clean-empty `emptyResult()` and skip `emptyResult({skipped:true})`. The degrade returns (`store(truncated)`→`created:1`, dry-run preview) stay EXPLICIT — collapsing them into the loop would misreport the count when the blob dedups.

## Store/insert internals
- `storeImmediate` (engine-impl) merges on cosine distance < EXACT_DEDUP_DISTANCE via `updateMemoryContent` (which stamps `updated_at = Date.now()`, and sets `importance = MAX(importance, ?)` — merges raise but never lower importance). Identical content → identical BGE embedding → distance 0 → deterministic exact-dedup merge (useful for tests).
- `insertMemory` (`storage/queries.ts`) stamps created/updated/last_accessed all to `Date.now()` unless `InsertMemoryParams.createdAt` is set; `source` was already a param but `storeImmediate` historically didn't forward it.

## Testing patterns
- Real-embedder tests: `mkdtempSync` + `initDatabase(dbPath, config.embeddingModel)` (loads BGE `Xenova/bge-base-en-v1.5`), 30s vitest timeout (`vitest.config.ts`). `loadConfig({})` gives defaults.
- To mock the LLM while letting the REAL embedder + store/dedup run: `vi.mock('../src/llm/client.js', …)` with `vi.hoisted(...)` state; mock both `llmComplete` (queue of canned responses) and `getLLMClient` (truthy stub vs null). The engine path imports `llmComplete` transitively via `storage/extraction.ts`, so mocking the module is sufficient.
- PII-safe assertion idiom: `vi.spyOn(console,'error')`, trigger failure with a sensitive marker in the blob, assert `errorSpy.mock.calls` joined never contains the marker, and the thrown/returned text doesn't either.

## memory-ingest design deviation (docs/designs/memory-ingest-tool.md v2)
- The design's §5.4 merge rule + §11 test expectation claim `updated_at === T_NEW` after a backdated merge. This is INTERNALLY INCONSISTENT: the merge runs `updateMemoryContent` (stamps `updated_at = now`) THEN `updateMemoryTimestampsOnMerge` doing `MAX(updated_at, asOf)`. With `asOf` in the past, `MAX(now, asOf) = now`, so `updated_at` lands at ~now, NOT T_NEW. The honest/faithful behavior: `updated_at = max(now, asOf)`; only `created_at` (MIN) and `last_accessed_at` (MAX, NOT touched by updateMemoryContent) reconcile cleanly to the as_of values. Implemented that way; tests assert the truthful invariant. Order-independence holds for created_at/last_accessed_at.
